動態/靜態/介面測試深入

Dynamic/Static/Interface Testing
01 — 中期 D6：軟體測試技術深入

今天深入軟體安全測試：SAST(靜態)、DAST(動態)、IAST 的互補，介面/API 測試，以及誤用案例與負向測試。與 D8 相通，但 D6 從『評估與測試』角度看。

🗺️ 今天重點：SAST vs DAST vs IAST → fuzzing → 介面/API 測試 → 正向 vs 負向/誤用案例。
02 — 測試技術比較
技術視角特點
SAST 靜態看原始碼(白箱)早期、找根因；誤報多
DAST 動態打執行中(黑箱)近真實、找執行期漏洞；較晚
IAST 互動執行中插樁結合兩者、較準
Fuzzing 模糊測試餵異常輸入找未處理的崩潰/邊界漏洞
03 — 介面與 API 測試
介面測試：驗系統間/模組間介面的輸入處理與錯誤處理
API 測試：驗授權(防越權 BOLA)、輸入驗證、速率限制、錯誤訊息不洩資訊
API 是現代主要攻擊面(微服務/行動後端)，須納入測試範圍
04 — 正向 vs 負向/誤用案例

正向測試驗『正常輸入功能正常』；負向/誤用案例測試從攻擊者視角驗『惡意/異常輸入會不會被擋』。只做正向會放過安全漏洞——安全 bug 往往藏在沒人測的異常路徑。fuzzing 是自動化的負向測試。

05 — 關鍵名詞速查
SAST（靜態應用測試）— 分析原始碼找漏洞(早期、白箱)。
DAST（動態應用測試）— 攻擊執行中應用(黑箱)。
IAST（互動式測試）— 執行中插樁、結合靜動態。
Fuzzing（模糊測試）— 餵大量異常輸入找崩潰/漏洞。
Misuse Case（誤用案例）— 攻擊者視角的負向測試。
API Testing（API 測試）— 驗授權/輸入/速率/錯誤處理。
Negative Testing（負向測試）— 驗異常/惡意輸入被正確拒絕。
06 — 易混淆釐清

SAST vs DAST：SAST 不執行、看原始碼(找得到行號、誤報多)；DAST 不看碼、打執行中應用(近真實、難定位根因)。互補非互斥。

正向 vs 負向：正向驗功能、負向(誤用/fuzzing)驗安全。只做正向會漏安全漏洞。

Fuzzing：自動化餵畸形/隨機輸入找未處理的例外與記憶體錯誤，特別擅長找未知邊界漏洞。

07 — 條文與標準

OWASP Testing Guide / ASVS / API Security Top 10、NIST SP 800-115。SAST/DAST/IAST/SCA 應整合進 CI/CD(D8 DevSecOps)，未過關卡不得部署。

08 — 真實事件：API 未測漏洞
🔎 前端沒給的，後端 API 卻給

常見漏洞：前端 UI 限制了，但API 未做伺服器端授權檢查——攻擊者直接呼叫 API、改一個 ID 就讀到別人的資料（BOLA/越權物件存取，OWASP API #1）。根因是API 未被獨立、負向地測試授權，只測了『正常使用』。多起行動 App 外洩源於此。

教訓：授權必須在伺服器端、每個物件都驗；API 要做負向/越權測試，不能假設前端會擋。前端限制不是安全控制。

📌 延伸：為何 SAST 誤報多卻仍重要

SAST 因缺乏執行期上下文常誤報(把不可達的路徑也報)，需人工分流。但它的價值在最早期、在開發者寫碼當下就抓到根因(行號)，修復成本最低(shift left)。實務是 SAST(早期廣篩)＋DAST(後期實證)＋人工審查＋SCA(相依)組合，各補盲點。

09 — CISSP 考點提醒

選最佳答案：『不執行、看原始碼』→ SAST；『打執行中應用』→ DAST；『餵異常輸入找崩潰』→ fuzzing；『驗惡意輸入被擋』→ 負向/誤用案例；『API 越權』→ 伺服器端授權＋負向測試。

常見陷阱：只做正向測試漏安全；前端限制非安全控制；SAST≠DAST(讀碼 vs 打站)。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：SAST 讀碼(早期)、DAST 打站、IAST 插樁、fuzzing 餵異常；API 要負向測授權(防 BOLA)；正向驗功能、負向驗安全。

Q1 . 自動餵畸形輸入找未處理崩潰的測試？
→ Fuzzing(模糊測試)，屬負向測試。

Q2 . 前端限制了但 API 仍可越權讀資料，根因？
→ API 未做伺服器端授權/未做負向測試(BOLA)；前端非安全控制。

Q3 . SAST 與 DAST 為何要併用？
→ SAST 早期看碼找根因(誤報多)、DAST 後期實證執行期漏洞，互補盲點。
