考點：事件管理流程順序

Exam: Incident Management Sequence
01 — 考前重點濃縮

D7 後期。本封考事件回應流程順序(最常考的排序題)與各階段目標。

NIST 順序:準備→偵測分析→遏制→根除→復原→事後檢討。
先遏制(止血)再根除(清病灶)再復原;順序錯會再次淪陷。
復原後要加強監控確認未復發;檢討轉成改善。
02 — 流程速記
準備→偵測分析→遏制→根除→復原→檢討
03 — 關鍵名詞（中英）
Preparation（準備）— 計畫/團隊/工具/演練。
Detection & Analysis（偵測分析）— 發現並研判事件。
Containment（遏制）— 止血、限制擴散。
Eradication（根除）— 清除病灶與根因。
Recovery（復原）— 恢復營運並加強監控。
Lessons Learned（事後檢討）— 回饋改善。
04 — 常見陷阱

遏制先於根除/復原:先止血再清病灶。

準備是第一步(平時做好);別跳過。

未根除就復原會再次淪陷。

05 — 🎯 實戰考題
QUESTION 1

According to NIST, what is the correct ORDER of the incident response phases (after Preparation)?

（依 NIST,事件回應各階段(在『準備』之後)的正確順序為何?）

A. Eradication → Containment → Recovery 根除→遏制→復原
B. Detection → Containment → Eradication → Recovery 偵測→遏制→根除→復原
C. Recovery → Containment → Eradication 復原→遏制→根除
D. Containment → Recovery → Eradication 遏制→復原→根除
✅ 解答與解析

正解：B。正確順序:偵測分析→遏制(止血)→根除(清病灶)→復原→檢討。先遏制再根除再復原。

為何其他選項錯：
  • A：遏制必須在根除『之前』(先止血)。
  • C：復原不能在遏制/根除之前(病灶未清會復發)。
  • D：復原不可在根除之前。
QUESTION 2

During an active malware outbreak, the FIRST priority after detection is to:

（在惡意程式正在擴散時,偵測後的第一要務是:）

A. Rebuild all systems 重建所有系統
B. Contain the incident to stop the spread 遏制事件以阻止擴散
C. Write the final report 撰寫最終報告
D. Identify lessons learned 整理經驗教訓
✅ 解答與解析

正解：B。偵測後第一要務是遏制(containment)——隔離受影響系統、阻止擴散(止血),之後才根除與復原。

為何其他選項錯：
  • A：重建屬復原階段,需先遏制與根除。
  • C／D：報告與檢討在事件處理後期。
QUESTION 3

Why must eradication occur BEFORE recovery?

（為何根除必須在復原『之前』?）

A. To save money 為了省錢
B. To ensure the threat is fully removed so systems are not reinfected 確保威脅完全移除,系統不會再次受感染
C. To speed up the process 加快流程
D. Regulations require it 法規要求
✅ 解答與解析

正解：B。若未根除(清除惡意碼/後門/被建帳號與根因)就復原上線,系統會再次被感染/淪陷。

為何其他選項錯：
  • A／C：非核心原因。
  • D：是技術必要性而非單純法規要求。
06 — (ISC)² 選最佳答案思路

順序題:準備→偵測→遏制→根除→復原→檢討。看到『正在擴散』先選 Containment。

根除必須在復原前(否則復發);準備是平時就要做的第一步。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：事件回應:準備→偵測→遏制→根除→復原→檢討;先遏制(止血)再根除(清病灶)再復原。

Q1 . 偵測後的第一要務(正在擴散時)？
→ 遏制(containment),阻止擴散。

Q2 . 根除為何在復原之前？
→ 確保威脅完全移除,避免復原後再次淪陷。

Q3 . 事件回應的第一個階段？
→ 準備(preparation,平時做好)。

📑 官方考綱對照重點（Official Objectives · §7.6 事件流程順序（考點））
▍步驟順序（高頻）

DRMRRRL：Detection → Response → Mitigation → Reporting → Recovery → Remediation → Lessons Learned（前置 Preparation）。

陷阱：Mitigation＝containment 控制損害，不是修好；Remediation 才做根因。絕不反擊攻擊者；涉 PII 須通報。

對應官方 §7.6（事件流程順序，考點）。

§7.6.1 Detection
• Detection: the identification of potential security incidents via monitoring and analyzing security logs, threat intelligence, or incident reports; as above, understanding the distinction between an event and an incident, the goal of detection is to identify an adverse event (an incident) and begin dealing with it
• Common methods to detect incidents:
• intrusion detection and prevention systems
• antimalware
• automated tools that scan audit logs looking for predefined events
• end users sometimes detect irregular activity and contact support
• Note: receiving an alert or complaint doesn’t always mean an incident has occurred
