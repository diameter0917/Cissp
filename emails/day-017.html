核心協定與安全協定

Core & Secure Protocols
01 — 協定：網路的共同語言

今天認識核心協定（TCP/UDP/ICMP）與安全協定（TLS/IPsec/SSH），以及最關鍵的 DNS 與 DNSSEC。初期抓「每個協定解什麼問題、安全版本是哪個」。

🗺️ 今天重點：TCP 三向交握 → 安全協定 TLS/IPsec/SSH → DNS 與 DNSSEC → 常見埠。
02 — 安全協定速覽
協定用途取代/強化
TLS加密傳輸（HTTPS）取代 SSL；保 in-transit
IPsecL3 VPN 加密（AH/ESP）站對站/遠端 VPN
SSH加密遠端登入/傳檔取代 Telnet/FTP
DNSSEC為 DNS 回應加簽章防 DNS 偽造/快取污染

💡 不安全→安全對照：Telnet→SSH、FTP→SFTP/FTPS、HTTP→HTTPS、SNMPv1/2→SNMPv3、DNS→DNSSEC。考題愛問「該換成什麼」。

03 — TCP 三向交握
SYN→SYN-ACK→ACK→連線建立

三向交握建立可靠連線；SYN flood 即濫發 SYN 不回 ACK 耗盡資源（L4 DoS），以 SYN cookie 緩解。

04 — 關鍵名詞速查
TLS（傳輸層安全）— 加密與驗證傳輸，HTTPS 基礎（埠 443）。
IPsec（IP 安全）— L3 加密；AH 驗完整、ESP 加密＋驗。
SSH（安全外殼）— 加密遠端管理與傳檔（埠 22）。
DNS（網域名稱系統）— 名稱↔IP 解析（埠 53）。
DNSSEC（DNS 安全擴充）— 以數位簽章驗 DNS 回應真實性與完整性。
ICMP（網際控制訊息）— 診斷/錯誤回報（ping）；常被濫用於掃描/洪水。
05 — 易混淆釐清

IPsec AH vs ESP：AH 只提供完整性/驗證（不加密）；ESP 提供加密＋完整性。需機密性就用 ESP。

DNSSEC 不等於加密：DNSSEC 提供真實性/完整性（防偽造），不加密查詢內容；要加密查詢是 DoH/DoT。

06 — 條文與標準

TLS（IETF RFC 8446＝TLS 1.3）、IPsec（RFC 4301）、SSH（RFC 4251）、DNSSEC（RFC 4033+）。常見埠：HTTP 80 / HTTPS 443 / SSH 22 / DNS 53 / SMTP 25 / RDP 3389——考題常考埠對應。

07 — 真實事件：DNS 劫持
🔎 把你導到假網站

DNS 劫持／快取污染（cache poisoning）讓使用者輸入正確網址，卻被解析到攻擊者的假伺服器，進而竊取憑證或植入惡意程式。根因是傳統 DNS 回應未經驗證。DNSSEC 以簽章讓解析器能驗證回應真偽，正是對策。

對照 Domain 1 的真實性（Authenticity）：DNSSEC 解決的是「我連到的真的是它嗎」。

08 — CISSP 考點提醒

選最佳答案：「加密遠端管理」→ SSH；「站對站 VPN」→ IPsec；「防 DNS 偽造」→ DNSSEC；「保網頁傳輸」→ TLS。

常見陷阱：AH 不加密（只 ESP 加密）；DNSSEC 不加密內容。Telnet/FTP/HTTP 為明文，應換安全版。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：核心協定 TCP（可靠）/UDP（快）/ICMP（診斷）；安全協定 TLS/IPsec/SSH 各有戰場；DNSSEC 用簽章防 DNS 偽造但不加密。

Q1 . IPsec 的 ESP 與 AH 差別？
→ ESP 加密＋完整性；AH 只完整性/驗證、不加密。

Q2 . DNSSEC 提供加密嗎？
→ 否；提供真實性/完整性（防偽造），加密查詢需 DoH/DoT。

Q3 . 取代明文 Telnet 的安全協定？
→ SSH（埠 22）。
