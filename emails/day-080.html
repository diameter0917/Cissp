外購軟體與惡意程式深入

Acquired Software & Malware Deep Dive
01 — 中期 D8 收尾：外購軟體與惡意程式

D8 與 P2 中期收尾,深入第三方元件風險治理、惡意程式類型與防禦、與軟體安全驗收。把『引入的軟體也要評估』做到位,並認識威脅本身。

🗺️ 中期收尾:D1–D8 已深入。下週進入後期 P3,回到 D1 聚焦考試重點與英文實戰考題。
02 — 第三方/外購軟體治理
採購前:供應商安全評估(SOC 2、滲透報告、SBOM)、合約安全條款與稽核權
導入:完整性驗證(簽章/雜湊)、安全驗收測試、最小權限部署
持續:相依漏洞監控(SCA)、修補追蹤、退出策略(避免 lock-in)
COTS/開源/SaaS 各有風險面(D1 P1-D8 已概覽,此處治理化)
03 — 惡意程式類型(深入)
類型特性
Virus/Worm病毒需宿主;蠕蟲自我傳播
Trojan/RAT偽裝;RAT 遠端控制
Ransomware加密勒索(常雙重勒索)
Rootkit/Bootkit隱藏自身、取得最高權限
Logic Bomb條件觸發
Fileless無檔案、藏記憶體/合法工具(LOLBins)
04 — 惡意程式防禦

分層防禦:EDR/XDR(行為偵測)取代純特徵防毒、應用白名單(allowlisting)、沙箱引爆、修補與最小權限、巨集/腳本管控、使用者意識。對無檔案/LOLBins 尤需行為偵測(特徵抓不到)。軟體驗收則防『一開始就引入惡意/有漏洞的軟體』。

05 — 關鍵名詞速查
COTS（商用現成軟體）— 黑箱、靠原廠修補。
Software Acceptance（軟體驗收）— 上線前驗安全與功能。
Worm（蠕蟲）— 自我複製傳播、不需宿主。
RAT（遠端存取木馬）— 攻擊者遠端控制主機。
Rootkit（隱匿程式）— 隱藏自身、最高權限。
Fileless Malware（無檔案惡意程式）— 藏記憶體/濫用合法工具。
Allowlisting（應用白名單）— 只允許核准程式執行。
06 — 易混淆釐清

Virus vs Worm:病毒需宿主且常需執行;蠕蟲自我傳播(WannaCry/Mirai)。

Trojan vs Virus:木馬偽裝誘騙安裝、不自我複製;病毒感染其他檔案。

特徵 vs 行為偵測:特徵抓已知;無檔案/多型/LOLBins 需行為偵測(EDR)。

Allowlist vs Blocklist:白名單(只允許核准)比黑名單(禁已知壞)更能擋未知惡意。

07 — 條文與標準

NIST SP 800-161(C-SCRM)、SP 800-83(惡意程式防護)、SP 800-167(應用白名單)、SSDF/SLSA(供應鏈)。軟體驗收對應變更管理與上線審查。

08 — 真實事件：第三方元件/供應鏈植入
🔎 信任的軟體,藏著後門

攻擊者把惡意碼植入受信任的軟體更新或熱門套件(SolarWinds 2020、惡意 npm/PyPI、被劫持的瀏覽器擴充),使用者正常安裝官方更新卻引入後門。因來源『可信』,傳統防護難察覺,影響面極廣。第三方元件的一個漏洞(如 Log4Shell)也會波及所有使用者。

教訓:外購/引入軟體不是『買來就安全』——需供應商評估、完整性驗證、SBOM、安全驗收、持續監控。收束整個 Domain 8 與供應鏈(D1)主題:信任要驗證(verify),不是預設(assume)。

📌 延伸：軟體驗收的安全關卡

上線前的安全驗收應包含:SAST/DAST/SCA 結果審查、滲透測試、設定基準檢查、SBOM 與授權合規、回退計畫。對外購軟體則加供應商安全證明、完整性驗證、最小權限部署。驗收是『把關引入點』的最後機會——通過才進正式環境,呼應變更管理(D7)。

09 — CISSP 考點提醒

選最佳答案：『自我傳播不需宿主』→ Worm;『偽裝誘騙安裝』→ Trojan;『藏記憶體/濫用合法工具』→ fileless(行為偵測);『只允許核准程式執行』→ allowlisting;『外購軟體上線前』→ 安全驗收＋完整性驗證。

常見陷阱：病毒(需宿主) vs 蠕蟲(自傳播);特徵抓不到無檔案(需 EDR);信任要驗證;白名單比黑名單強。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：外購軟體要評估/驗收/完整性驗證/SBOM/持續監控;惡意程式分病毒(需宿主)/蠕蟲(自傳播)/木馬/勒索/rootkit/fileless;防禦靠 EDR行為偵測＋白名單＋最小權限。

Q1 . 蠕蟲與病毒最大差別？
→ 蠕蟲自我傳播、不需宿主;病毒需宿主檔案。

Q2 . 為何無檔案惡意程式難用傳統防毒擋？
→ 它藏在記憶體/濫用合法工具(LOLBins)、無檔案特徵;需行為偵測(EDR)。

Q3 . 引入外購/開源軟體的關鍵把關控制？
→ 供應商評估、完整性驗證(簽章/雜湊)、SBOM/SCA、上線安全驗收、持續監控。

📑 官方考綱對照重點（Official Objectives · §8.4 外購軟體與惡意程式（深入））
▍各類外購評估

COTS：查已知漏洞處理態度、ISO/IEC 27034/27001、FedRAMP。Open source：可審程式但須自負支援與更新（怕過時函式庫）。Third-party：合約綁外部稽核、通過才付款。

▍供應鏈

SBOM 盤點所有元件/依賴；雲端（SaaS/IaaS/PaaS）外洩主因＝錯誤設定/存取控制不良；XaaS 泛指各類雲服務。

對應官方 §8.4 外購軟體（深入）。

§8.4.1 Commercial-off-the-shelf (COTS)
• Commercial Off-the-Shelf (COTS): software elements, usually apps, that are provided as finished products (not intended for alteration by or for the end-user)
• Most widely used COTS software products have been tested by security researchers (both benign and malicious)
• researching discovered vulnerabilities and exploits can help us understand how seriously the vendor takes security
• for niche products, you should research vendor certifications, such as ISO/IEC 27034 Application Security
• other than secure coding certification, you can look for overall information security management system (ISMS) certifications such as ISO/IEC 27001 and FedRAMP (which are difficult to obtain, and show that the vendor is serious about security)
• If you can talk with a vendor, look for processes like defensive programming, which is a software development best practice that means as code is developed or reviewed, they are constantly looking for opportunities for things to go badly
• e.g. treating all input routines as untrusted until proven otherwise
§8.4.3 Third-party
• Third-party software: (AKA outsourced software) is software made specifically for an org by a third party
• third-party software is not considered COTS, since the software is custom or customized
• third-party software may rely on open-source software, but since it's customized, it may have different or additional vulns
• it's best practice to use a third-party to do an external audit and security assessment; this should be built into the vendor's contract, with passing the audit conditional for finalizing software purchase
