Domain 5 全景與 AAA

Domain 5 Big Picture & AAA
01 — Domain 5 與 AAA

Domain 5（Identity and Access Management，身分與存取管理）佔 13%，是第一道防線——絕大多數入侵都從「身分被冒用」開始。核心框架是 IAAA：識別、鑑別、授權、可歸責。今天先把這四個動作分清楚。

🗺️ 本週（Domain 5 · 初期）路線：①今天 AAA 全景 → ②鑑別因子與 MFA → ③身分生命週期與聯合 → ④聯合協定 → ⑤存取控制模型與 PAM。
02 — IAAA 四步驟
Identification 識別→Authentication 鑑別→Authorization 授權→Accountability 可歸責
步驟回答的問題例子
Identification 識別你宣稱是誰？輸入帳號/使用者名稱
Authentication 鑑別你能證明嗎？密碼/OTP/指紋
Authorization 授權你可以做什麼？依角色/權限決定可存取資源
Accountability 可歸責誰做了什麼？稽核日誌可追溯到個人

💡 可歸責的前提是「每人唯一身分＋完整日誌」。共用帳號＝無法歸責，是常見稽核缺失。

03 — 關鍵名詞速查
Identification（識別）— 宣稱身分（提供 identifier，如帳號）。
Authentication（鑑別）— 證明身分（提供 factor，如密碼）。
Authorization（授權）— 決定身分可存取哪些資源與動作。
Accountability（可歸責）— 行為可追溯到特定個人（靠唯一身分＋日誌）。
IAM（身分與存取管理）— 管理身分生命週期與存取的整體框架。
Non-repudiation（不可否認）— 結合鑑別與日誌，使行為人無法否認其作為。
04 — 易混淆釐清

Identification vs Authentication：識別是「宣稱我是誰」（帳號）；鑑別是「證明」（密碼/生物特徵）。帳號本身不是祕密、不算鑑別。

Authentication vs Authorization：鑑別＝你是誰；授權＝你能做什麼。通過鑑別不代表有權限。

05 — 條文與標準

NIST SP 800-63（Digital Identity）是身分鑑別的權威，定義 IAL/AAL/FAL 等級（中期深入）。可歸責也是各稽核框架（ISO 27001、SOC 2）的基本要求。

06 — 真實事件：缺可歸責，無法追責
🔎 共用帳號的代價

當團隊共用一組管理員帳號（如 admin/root），一旦發生誤刪或惡意操作，日誌只顯示「admin 做的」——無法追溯到實際個人，調查與究責全部卡住。這是稽核中極常見的缺失，也讓不可否認性蕩然無存。

對策：每人唯一身分、特權操作走 PAM（Day 25）、完整且受保護的稽核日誌。

07 — CISSP 考點提醒

選最佳答案：「輸入帳號」＝Identification；「輸入密碼/指紋」＝Authentication；「能否存取某檔」＝Authorization；「追溯誰做的」＝Accountability。把動作對應到 IAAA。

常見陷阱：共用帳號破壞可歸責；識別（帳號）不是鑑別。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：IAM 是第一道防線；IAAA＝識別→鑑別→授權→可歸責，可歸責需唯一身分＋日誌。

Q1 . 輸入使用者名稱屬於 IAAA 哪一步？
→ Identification 識別（宣稱身分）。

Q2 . 為何共用帳號是稽核缺失？
→ 破壞 Accountability，無法追溯到個人、喪失不可否認性。

Q3 . 通過鑑別就等於有權限嗎？
→ 否；授權（Authorization）另外決定能做什麼。

🎯 實戰演練 EXTRA
QUESTION 1

Within the IAAA framework, which access control service defines the level of access an identified and authenticated subject has to an object?

（在 IAAA 框架中,哪一項存取控制服務定義了已識別且已鑑別的主體對物件可擁有的存取層級?）

A. Identification 識別
B. Authentication 鑑別
C. Accountability 可歸責
D. Authorization 授權
✅ 解答與解析

正解：D。Authorization 授權回答『你可以做什麼』——依已證明的身分,定義主體對資源的存取層級與可執行的操作。通過鑑別只證明『你是誰』,不代表你有任何權限。記憶:AuthN＝who you are,AuthZ＝what you can do。

為何其他選項錯：
  • A：Identification 只是宣稱身分（提供帳號等 identifier）,不決定存取層級。
  • B：Authentication 驗證身分真偽,但不定義能存取什麼。
  • C：Accountability 是事後透過日誌追溯行為,非事前定義存取。
QUESTION 2

A security manager must ensure that employees who approve high-value wire transfers cannot later deny having performed the approval. Which combination of controls BEST supports this requirement?

（安全經理必須確保核准高額電匯的員工,事後無法否認其核准行為。哪一組控制最能支持此需求?）

A. A shared approver account protected by a strong password 以強密碼保護的共用核准帳號
B. Unique individual identities combined with authentication and protected audit logs 唯一個人身分,結合鑑別與受保護的稽核日誌
C. Encrypting all transfer records at rest 對所有電匯紀錄做靜態加密
D. Requiring approvers to change passwords every 30 days 要求核准者每 30 天更換密碼
✅ 解答與解析

正解：B。不可否認性（non-repudiation）＝鑑別＋可歸責:每人有唯一身分、通過鑑別、行為寫入受保護的稽核日誌,行為人才無法否認其作為。這正是 IAAA 中 Accountability 要達成的管理目的。記憶:唯一身分＋完整日誌＝可歸責。

為何其他選項錯：
  • A：共用帳號正好破壞可歸責——日誌無法對應到個人,是稽核大忌。
  • C：靜態加密保護機密性,無法證明『誰』做了核准。
  • D：密碼輪換屬鑑別衛生,與能否否認行為無關。
📑 官方考綱對照重點（Official Objectives · §5.2.2 AAA）
▍四大存取控制服務

Identification 識別（主張身分）→ Authentication 鑑別（驗證身分）→ Authorization 授權（定義存取）→ Accountability 當責（靠 Auditing 稽核達成）。

▍三大鑑別因子

知識 something you know（密碼）／持有 something you have（token）／特徵 something you are（生物特徵）。MFA 須不同類型因子；FIDO2/WebAuthn＝防釣魚的無密碼鑑別。

對應官方 §5.2.2 AAA 與 MFA。

§5.2.2 Authentication, Authorization and Accounting (AAA) (e.g., multi-factor authentication (MFA), password-less authentication)
• The four key access control services: Identification (assertion of identity), Authentication (verification of identity), Authorization (definition of access), Accountability (responsibility of actions)
• Note that AAA is the same principles of Authentication, Authorization, and using the word Accounting instead of Accountability (but it's the same principle)
• and remember that the three factors of authentication that you need to understand is knowledge, ownership, and characteristic (see above)
• Two important security elements in an access control system are authorization and accountability
• Authorization: subjects are granted access to objects based on proven identities; the level of access defined for the identified and authenticated user or process
• Accountability AKA Principle of Access Control: proper identification, authentication, and authorization that is logged and monitored; users and other subjects can be held accountable for their actions when auditing is implemented; accountability is maintained for individual subjects through the use of auditing; logs record user activities and users can be held accountable for their logged actions; this encourages good user behavior and compliance with the org's security policy; also see definitions/interpolations in Domain 2, and above
• Auditing: tracks subjects and records when they access objects, creating an audit trail in one or more audit logs
• Auditing provides accountability
• FIDO2 and WebAuthn are related technical standards that enable phishing-resistant, passwordless authentication across the internet
• FIDO2 (Fast IDentity Online 2): an overarching project and set of specifications developed by the FIDO Alliance designed to replace passwords with cryptographic keys
• WebAuthn (Web Authentication): a specific web API standard developed by the World Wide Web Consortium (W3C) in collaboration with FIDO that allows websites (known as "Relying Parties") to communicate directly with a user’s browser to perform secure authentication
• Single-factor authentication: any authentication using only one proof of identity
• Two-factor authentication (2FA): requires two different proofs of identity
• Multifactor authentication (MFA): any authentication using two or more factors
• multifactor auth must use multiple types or factors, such as something you know and something you have
• note: requiring users to enter a password and a PIN is NOT multifactor (both are something you know)
• Two-factor methods:
• Hash Message Authentication Code (HMAC): includes a hash function used by the HMAC-based One-Time Password (HOTP) standard to create onetime passwords
• Time-based One-Time Password (TOTP): similar to HOTP, but uses a timestamp and remains valid for a certain time frame (e.g. 30 or 60 seconds)
• e.g. phone-based authenticator app, where your phone is mimicking a hardware TOTP token (combined with userid/password is considered two-factor or two-step authentication)
• Email challenge: popular method, used by websites, sending the user an email with a PIN
• Short Message Service (SMS): to send users a text with a PIN is another 2-factor method; note that NIST SP 800-63B points out vulnerabilities, and deprecates use of SMS as a two-factor method for federal agencies
• Password-less authentication: a method of verifying a user's identity without requiring them to enter a password; uses alternate verification forms like biometrics, security tokens, or mobile device
• this is an important topic, because password use (and misuse) provide many security headaches and problems
• Advantages of password-less auth include:
• increased security
• improved user convenience
• reduction risk of phishing: if attacker gains access to a password, but password-less auth makes it much more difficult for the attacker to access the associated device (say if password-less auth is via mobile device)
• Disadvantages of password-less auth:
• dependency on devices (e.g. if via mobile phone, that device is required for access)
• biometric issues associated with reliability and privacy
• implementation costs associated with additional hardware devices etc
