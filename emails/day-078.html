注入/XSS/CSRF 與安全編碼深入

Injection/XSS/CSRF & Secure Coding
01 — 中期 D8：常見漏洞與安全編碼深入

今天深入注入(SQLi/命令)、XSS、CSRF 的機制與防禦,以及安全編碼原則:輸入驗證、輸出編碼、參數化、最小權限。理解攻擊原理才知道為何這些防禦有效。

🗺️ 今天重點:注入(SQLi/命令) → XSS 三型 → CSRF → 安全編碼黃金原則。
02 — 注入攻擊
SQL Injection:輸入被當 SQL 執行→繞登入/匯出資料庫;防:參數化查詢/預備語句＋輸入驗證＋DB 最小權限
Command Injection:輸入被當 OS 指令;防:避免呼叫 shell、白名單、參數化 API
共通根因:把『資料』當成『指令/程式』執行——讓資料永遠是資料即可根除
03 — XSS 三型與 CSRF
漏洞本質防禦
Stored XSS惡意腳本存進 DB,他人瀏覽時執行輸出編碼＋CSP
Reflected XSS腳本經 URL 反射執行輸出編碼＋輸入驗證
DOM XSS前端 JS 不安全處理 DOM安全 DOM API、避免 innerHTML
CSRF冒用已登入身分送請求CSRF token＋SameSite cookie

💡 一句話:XSS＝在受害者瀏覽器執行攻擊者腳本(利用網站對使用者輸出的信任);CSRF＝冒用使用者身分送請求(利用網站對使用者 cookie 的信任)。防法不同。

04 — 安全編碼黃金原則
不信任任何輸入:白名單驗證(validate)優於黑名單
參數化:讓資料不被當指令(擋注入)
輸出編碼:依情境(HTML/JS/URL/SQL)編碼(擋 XSS)
最小權限＋失敗安全(fail securely)＋深度防禦＋避免明文機密
05 — 關鍵名詞速查
SQL Injection（SQL 注入）— 輸入被當 SQL 執行。
Command Injection（命令注入）— 輸入被當 OS 指令執行。
XSS（跨站腳本）— 在他人瀏覽器執行腳本。
CSRF（跨站請求偽造）— 冒用已登入身分送請求。
Input Validation（輸入驗證）— 白名單檢核輸入。
Output Encoding（輸出編碼）— 依情境編碼中和腳本。
CSP（內容安全政策）— 限制可載入的腳本來源(緩解 XSS)。
Parameterized Query（參數化查詢）— 資料與指令分離,根除注入。
06 — 易混淆釐清

XSS vs CSRF:XSS 在受害者瀏覽器執行腳本(竊 cookie/session);CSRF 冒用身分送請求(改密碼/轉帳)。XSS 靠輸出編碼/CSP、CSRF 靠 token/SameSite。

輸入驗證 vs 輸出編碼:驗證擋不合法輸入;編碼讓資料在呈現/使用時不被當程式。兩者都要,不可互相取代。

黑名單 vs 白名單:黑名單(禁特定壞字)易繞過;白名單(只允許已知好)較可靠。

07 — 條文與標準

OWASP Top 10 / ASVS / Cheat Sheets、CWE(弱點分類)、SEI CERT 安全編碼標準。CSP=W3C;SameSite cookie 已成預設防 CSRF 基線。

08 — 真實事件：XSS 竊取 session
🔎 一段腳本,接管你的帳號

Stored XSS 經典危害:攻擊者把惡意腳本存入留言/個資欄位,當其他使用者(含管理員)瀏覽時,腳本在他們的瀏覽器執行,竊取 session cookie 傳給攻擊者→接管帳號(無需密碼)。多起社群/論壇平台曾因此被蠕蟲式擴散。

教訓:輸出編碼＋CSP＋HttpOnly cookie(讓 JS 讀不到 session)＋輸入驗證多層防禦。HttpOnly 讓即使有 XSS 也偷不到 session cookie,是重要的縱深控制。

📌 延伸：為何輸入驗證不足以擋 XSS

只在輸入端驗證不夠:同一份資料可能用在多種輸出情境(HTML body、屬性、JS、URL),每種需不同編碼。且資料可能來自其他來源(DB、API)未經你的輸入驗證。故防 XSS 的主力是『在輸出時依情境編碼』,輸入驗證為輔。這是常見觀念錯誤。

09 — CISSP 考點提醒

選最佳答案：『防 SQLi 最根本』→ 參數化查詢;『防 XSS 主力』→ 輸出編碼(＋CSP/HttpOnly);『防 CSRF』→ token/SameSite;『輸入驗證用』→ 白名單。

常見陷阱：XSS(執行腳本) vs CSRF(冒用請求)防法不同;輸出編碼是防 XSS 主力(非僅輸入驗證);黑名單不可靠。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：注入靠參數化(資料≠指令)、XSS 靠輸出編碼＋CSP＋HttpOnly、CSRF 靠 token＋SameSite;輸入用白名單驗證、輸出依情境編碼。

Q1 . 防 XSS 的主力控制是什麼、為何不只靠輸入驗證？
→ 依情境的輸出編碼;同資料用於多輸出情境且可能來自其他來源,輸入驗證為輔。

Q2 . XSS 與 CSRF 的防禦分別是？
→ XSS:輸出編碼/CSP/HttpOnly;CSRF:CSRF token/SameSite cookie。

Q3 . HttpOnly cookie 如何降低 XSS 危害？
→ 讓 JavaScript 讀不到 session cookie,即使有 XSS 也難竊取 session。

▍漏洞與防禦對照

SQLi→參數化查詢/預備語句、最小權限 DB 帳號。XSS（reflected/stored/DOM）→輸出編碼、CSP。CSRF→anti-CSRF token、SameSite cookie。共同：輸入驗證（白名單）、視輸入為不可信。

▍標準 vs 指引

Standards 標準＝強制、可跨全組織執行、可驗證（降漏最有效）；Guidelines 指引＝建議、保留彈性。Parameter validation 防 API 濫用。

對應官方 §8.5 安全編碼。

§8.5.1 Security weaknesses and vulnerabilities at the source-code level
• A source code vulnerability is a code defect providing a threat actor with an opportunity to compromise the security of a software system
• source code vulns are caused by design or implementation flaws
• design flaw: if dev did everything correctly, there would still be a vulnerability
• implementation flaw: dev incorrectly implemented part of a good design
• the OWASP top 10 vulnerabilities (2025 edition):
• Broken access control
• Security misconfiguration
• Software supply chain failures
• Cryptographic failures
• Injection
• Insecure design
• Authentication failures
• Software or data integrity failures
• Security logging and alerting failures
• Mishandling of exceptional conditions
§8.5.3 Security coding practices
• Secure coding practices can be summarized as standards and guidelines
• standards: mandatory activities, actions, or rules
• guidelines: recommended actions or ops guidelines that provide flexibility for unforeseen circumstances
• orgs greatly reduce source code vulns by enforcing secure coding standards and maintaining coding guidelines that reflect best practices
• To be considered a standard, coding practice must meet the following:
• reduces the risk of a particular type of vuln
• enforceable across all of an org's software development efforts
• verifiably implemented
• Note: secure coding standards, rigorously applied, is the best way to reduce source code vulns; coding standards ensure devs always do certain things in a certain way, while avoiding others
• Secure coding guidelines are recommended practices that tend to be less specific than standards
• e.g. consistently formatted code comments, or keeping code functions short/tight
