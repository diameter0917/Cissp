安全模型輪廓

Security Models Overview
01 — 安全模型：把安全規則「數學化」

安全模型（Security Models）把抽象的安全政策變成可驗證的規則。初期不必記死每條公理，先抓「每個模型在保護 C 還是 I、它的招牌規則是什麼」。考試最愛問「no read up / no write down」這類規則歸屬。

🗺️ 今天重點：Bell-LaPadula（機密）→ Biba（完整）→ Clark-Wilson（完整/交易）→ Brewer-Nash（利益衝突）。

💡 記憶法：BLP 顧機密（怕往下洩）→ no write down；Biba 顧完整（怕往上汙染）→ no write up。兩者規則正好相反。

03 — Bell-LaPadula 的兩條 + 一條
Simple Security Property：no read up（不能讀比自己高的密級）
* (star) Property：no write down（不能寫到比自己低的密級，防洩漏）
Strong * Property：只能在同級讀寫
04 — 關鍵名詞速查
Bell-LaPadula (BLP)（機密性模型）— no read up / no write down，軍方分級系統基礎。
Biba（完整性模型）— no read down / no write up，防完整性被低可信來源污染。
Clark-Wilson（完整性模型）— 透過 well-formed transaction 與職責分離保完整。
Brewer-Nash（利益衝突模型）— Chinese Wall，依歷史動態隔離競爭資料。
Reference Monitor（參考監視器）— 所有存取的仲裁者，須 tamperproof、always invoked、verifiable。
Security Kernel（安全核心）— 實作 reference monitor 的硬/韌/軟體核心。
05 — 易混淆釐清

BLP vs Biba：BLP＝機密（防往下洩漏，no write down）；Biba＝完整（防往上污染，no write up）。看到「保密/分級」想 BLP，看到「資料正確性/防竄改」想 Biba。

模型 vs 標準：模型（BLP/Biba）是理論規則；Common Criteria/EAL 是評估產品的標準（中期深入）。

06 — 條文與標準

這些模型源於 1970–80 年代美國國防研究；現代產品安全性則用 Common Criteria（ISO/IEC 15408）以 EAL1–7 評級（中期 P2-D3-1 深入）。Reference Monitor 概念仍是作業系統存取控制設計的基石。

07 — 真實事件：軍事分級系統
🔎 為什麼軍方要 no write down

在多密級（multilevel）系統中，一名持有 Top Secret 權限的人員若能把資料「寫到」Unclassified 區，就可能（無論有意或惡意程式所為）把機密降級洩漏。Bell-LaPadula 的 no write down（*-property）正是為防這種洩漏而生——這也是它成為軍事分級系統理論基礎的原因。

反過來，銀行帳務系統更怕「資料被錯誤寫入污染」，那是 Biba 與 Clark-Wilson 的守備範圍。

08 — CISSP 考點提醒

高頻考點：給規則問模型——「no write down」→ BLP（機密）；「no write up」→ Biba（完整）。先判斷題目在乎 C 還是 I 就能秒選。

常見陷阱：把 BLP 與 Biba 規則對調是最常見錯誤；Clark-Wilson 的重點是「使用者不能直接改資料，須透過程式」＋職責分離。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：BLP 顧機密（no read up / no write down）、Biba 顧完整（no read down / no write up）、Clark-Wilson 用交易與職責分離保完整、Brewer-Nash 擋利益衝突。

Q1 . 「no write down」屬於哪個模型？保護什麼？
→ Bell-LaPadula；保護機密性 Confidentiality。

Q2 . 想防止低可信來源污染高完整資料，用哪個模型？
→ Biba（no write up / no read down）。

Q3 . Clark-Wilson 的核心機制？
→ Well-formed transactions ＋ 職責分離；使用者須透過程式存取資料。
