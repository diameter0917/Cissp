安全協定與 DNS 深入

Secure Protocols & DNS Deep Dive
01 — 中期 D4：把通道加密與 DNS 守住

今天深入三大安全協定：TLS（握手與降級防護）、IPsec（AH/ESP 與模式）、以及 DNS/DNSSEC。這些是『讓不安全的網路變可信』的關鍵機制，也是高頻考點。

🗺️ 今天重點：TLS 握手與版本 → IPsec AH/ESP/傳輸 vs 通道 → DNS 攻擊 → DNSSEC。
02 — TLS 握手與安全
握手：協商版本/套件 → 驗證憑證（PKI）→ 用非對稱換對稱會談金鑰 → 對稱加密通訊（混合加密）
用 TLS 1.2/1.3；SSL 與 TLS 1.0/1.1 已淘汰
PFS：用 ECDHE 臨時金鑰，私鑰外洩也不影響過去會談
降級攻擊：誘使雙方退回弱版本/套件；對策：禁用舊版、TLS_FALLBACK_SCSV
03 — IPsec：AH vs ESP、傳輸 vs 通道
元件提供
AH完整性＋來源鑑別（不加密）
ESP機密性(加密)＋完整性（常用）
Transport 模式只加密 payload（端對端）
Tunnel 模式加密整個封包（site-to-site VPN）

💡 記憶：要加密用 ESP（AH 不加密）；站對站 VPN 用 Tunnel 模式（整包加密）。IPsec 在 L3。

04 — DNS 與 DNSSEC

DNS把網域轉 IP，缺乏驗證故易被快取下毒（cache poisoning）/spoofing導向釣魚站。DNSSEC 用數位簽章為 DNS 回應提供來源鑑別與完整性（但不加密、不提供機密性）。加密查詢另有 DoH/DoT。

05 — 關鍵名詞速查
TLS（傳輸層安全）— 握手＋混合加密，保護通道。
PFS（完美前向保密）— 臨時金鑰，私鑰洩漏不殃及過去。
Downgrade Attack（降級攻擊）— 誘使退回弱協定/套件。
IPsec AH/ESP（驗證標頭/封裝安全酬載）— AH 驗證不加密、ESP 加密。
Transport/Tunnel（傳輸/通道模式）— 加 payload vs 加整包。
DNSSEC（DNS 安全擴充）— 簽章保 DNS 完整性與來源(不加密)。
DoH/DoT（加密 DNS）— DNS over HTTPS/TLS，保查詢機密。
06 — 易混淆釐清

AH vs ESP：AH 只驗證（不加密）；ESP 可加密＋驗證。要機密性一定用 ESP。

Transport vs Tunnel：傳輸模式只保護 payload（端對端）；通道模式保護整個原始封包（VPN 閘道間）。

DNSSEC ≠ 加密：DNSSEC 提供完整性/來源鑑別（防 spoofing），不提供機密性；要加密查詢用 DoH/DoT。

SSL vs TLS：SSL 已全面淘汰；用 TLS 1.2/1.3。

07 — 條文與標準

TLS＝RFC 8446(1.3)、IPsec＝RFC 4301/4302(AH)/4303(ESP)、DNSSEC＝RFC 4033-4035、DoH＝RFC 8484。NIST SP 800-52（TLS 配置）、SP 800-77（IPsec/VPN）。

08 — 真實事件：TLS 降級攻擊
🔎 POODLE 與強迫降級

POODLE（2014）等攻擊利用『相容舊版』機制，強迫瀏覽器與伺服器退回到有漏洞的 SSL 3.0，再利用其弱點竊取資料。根因是系統為相容而保留弱協定。對策：停用 SSL/TLS 1.0-1.1、禁不安全套件、啟用降級保護。

教訓：『為了相容而保留弱選項』本身就是漏洞。安全配置要主動淘汰弱協定，不能依賴對方選好的。

📌 延伸：憑證釘選與 HSTS

為強化 TLS：HSTS 強制瀏覽器只用 HTTPS（防 SSL stripping 降級到 HTTP）；Certificate Pinning（憑證釘選）讓 App 只信任特定憑證/公鑰，防惡意 CA 簽發的假憑證做 MITM。但釘選需謹慎管理輪替，否則換憑證會導致服務中斷。

09 — CISSP 考點提醒

選最佳答案：『IPsec 要機密性』→ ESP；『站對站 VPN』→ Tunnel 模式；『防 DNS spoofing』→ DNSSEC；『加密 DNS 查詢』→ DoH/DoT；『防降級』→ 停用弱協定。

常見陷阱：AH 不加密；DNSSEC 不加密（只驗證）；SSL 已淘汰；降級攻擊源於保留弱版本。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：TLS 混合加密保通道、禁弱版防降級；IPsec ESP 加密、Tunnel 保整包；DNSSEC 驗證 DNS(不加密)、DoH/DoT 才加密。

Q1 . IPsec 中要加密必須用 AH 還 ESP？
→ ESP（AH 只驗證、不加密）。

Q2 . DNSSEC 提供機密性嗎？
→ 否；只提供完整性與來源鑑別（防 spoofing）。加密用 DoH/DoT。

Q3 . 站對站 VPN 用 IPsec 哪種模式？
→ Tunnel 模式（加密整個原始封包）。

🎯 實戰演練 EXTRA
QUESTION 1

During a penetration test, an organization's public web server is found to still accept SSL 3.0 and TLS 1.0 connections that were kept for compatibility with older clients. The tester demonstrates that an on-path attacker can force browsers to negotiate the weakest protocol. Which of the following is the BEST remediation?

（滲透測試發現,組織的對外 Web 伺服器為相容舊用戶端仍接受 SSL 3.0 與 TLS 1.0 連線。測試員展示路徑上的攻擊者可強迫瀏覽器協商到最弱協定。下列何者是最佳補救措施?）

A. Disable SSL 3.0/TLS 1.0-1.1 and weak cipher suites on the server, and enable downgrade protection such as TLS_FALLBACK_SCSV 在伺服器停用 SSL 3.0/TLS 1.0-1.1 與弱加密套件,並啟用 TLS_FALLBACK_SCSV 等降級保護
B. Replace the server certificate with one using a longer RSA key 更換為使用更長 RSA 金鑰的伺服器憑證
C. Require users to upgrade their browsers before connecting 要求使用者連線前先升級瀏覽器
D. Migrate the website from TLS to an IPsec tunnel 將網站由 TLS 移轉為 IPsec 通道
✅ 解答與解析

正解：A。降級攻擊（如 POODLE）的根因是『為相容而保留弱協定』——只要伺服器還接受 SSL 3.0/TLS 1.0,攻擊者就能強迫協商到弱版本再利用其漏洞。最佳補救是直接消除根因:停用弱版本與弱套件,並啟用 TLS_FALLBACK_SCSV 等降級保護。判斷邏輯:修補『根因』永遠優於補強周邊環節。記憶法:『保留弱選項,本身就是漏洞』。

為何其他選項錯：
  • B：SSL 3.0 的弱點在協定設計本身,金鑰長度再長也擋不住降級後的協定層攻擊。
  • C：依賴使用者行為不可控;只要伺服器仍接受弱協定,攻擊者就能誘導降級。
  • D：IPsec 不適用一般大眾瀏覽器存取的公開網站,且換協定並未處理『保留弱版本』的根因。
QUESTION 2

A network architect must protect routing protocol updates exchanged between two internal routers. Compliance requires source authentication and integrity so that forged updates are rejected, but the packet contents must remain readable for troubleshooting, and encryption is explicitly NOT required. Which IPsec option BEST meets these requirements?

（網路架構師須保護兩台內部路由器間交換的路由協定更新。法遵要求來源鑑別與完整性以拒絕偽造更新,但封包內容須保持可讀以利除錯,且明確不需要加密。哪個 IPsec 選項最符合需求?）

A. ESP in tunnel mode ESP 通道模式
B. ESP in transport mode ESP 傳輸模式
C. AH in transport mode AH 傳輸模式
D. TLS sessions between the routers 於路由器之間改用 TLS 會談
✅ 解答與解析

正解：C。先列需求再對映元件:要『來源鑑別＋完整性』且『不加密、內容可讀』——這正是 AH（Authentication Header）的定義:提供鑑別、完整性與防重放,但不加密。兩台路由器端對端通訊用 transport 模式即可。判斷邏輯:CISSP 常反向出題——大家記得『要加密選 ESP』,這題考的是『明確不要機密性』時,AH 才是精準選擇,ESP 反而超出需求。記憶法:AH 像『官方認證章』,蓋章但不封口,誰都能讀。

為何其他選項錯：
  • A：ESP 會加密 payload,違反『內容須可讀』的需求;tunnel 模式是閘道對閘道用,場景也不符。
  • B：模式正確但元件錯:ESP 仍會加密內容,不符『明確不需加密』的條件。
  • D：TLS 保護的是 TCP 應用會談,不適合 L3 路由協定更新的逐封包保護。
QUESTION 3

Users at a financial services firm report being redirected to a fraudulent website even though they typed the correct URL. Investigation shows the internal recursive resolver's cache contains forged records for the bank's domain. Which control would BEST prevent this attack from succeeding in the future?

（金融服務公司的使用者回報,即使輸入正確網址仍被導向詐騙網站。調查發現內部遞迴解析器的快取存有該銀行網域的偽造紀錄。哪個控制最能防止此攻擊再次得逞?）

A. Encrypt all DNS queries with DNS over HTTPS (DoH) 以 DNS over HTTPS（DoH）加密所有 DNS 查詢
B. Block outbound UDP port 53 at the perimeter firewall 在邊界防火牆封鎖出站 UDP 53 埠
C. Enable DNSSEC validation so DNS responses are verified with digital signatures 啟用 DNSSEC 驗證,以數位簽章驗證 DNS 回應
D. Increase the TTL of cached DNS records 增加 DNS 快取紀錄的 TTL
✅ 解答與解析

正解：C。症狀（正確網址被導向假站、快取有偽造紀錄）就是 DNS 快取毒化（cache poisoning）,根因是傳統 DNS 回應缺乏驗證。DNSSEC 以數位簽章提供來源鑑別與完整性,偽造紀錄因簽章驗不過而被拒收。判斷邏輯:攻擊利用『無驗證』,對策就是『加上驗證』;注意 DNSSEC 提供的是完整性/鑑別,不是加密。記憶法:DNSSEC＝『簽章防偽,不封口』。

為何其他選項錯：
  • A：DoH 加密的是查詢通道（機密性）,不驗證紀錄本身真偽,解析器快取仍可能被毒化。
  • B：封鎖 53 埠會癱瘓正常名稱解析,是斷服務而非控制措施。
  • D：拉長 TTL 反而讓已被毒化的紀錄在快取中存留更久,惡化問題。

▍IPSec 元件

SA（安全關聯，單向會談）、IKE/ISAKMP（金鑰協商，含 OAKLEY+SKEME）。AH vs ESP、transport vs tunnel 再複習。WireGuard＝現代精簡 VPN。

▍DNS 安全

DNS（UDP/TCP 53）；DNSSEC 以數位簽章防快取毒化/竄改（提供來源鑑別與完整性，不提供機密性）。

對應官方 §4.1.3 安全協定（IPSec/IKE 深入）。

§4.1.3 Secure protocols (e.g., Internet Protocol Security (IPSec), Secure Shell (SSH), Secure Sockets Layer (SSL)/Transport Layer Security (TLS))
• Kerberos: standards-based network authentication protocol, used in many products (most notably Microsoft Active Directory Domain Services or AD DS)
• Kerberos is mostly used on LANs for organization-wide authentication, single sign-on (SSO) and authorization
• SSL and TLS: data protection; used for protecting website transactions (e.g. banking, e-commerce)
• SSL and TLS both offer data encryption, integrity and authentication
• TLS has supplanted SSL (the original protocol, considered legacy/insecure)
• TLS was initially introduced in 1999 but didn’t gain widespread use until years later
• The original versions of TLS (1.0 and 1.1) are considered deprecated and organizations should be relying on TLS 1.2 or 1.3
• The defacto standard for secure web traffic is HTTP over TLS, which relies on hybrid cryptography: using asymmetric cryptography to exchange an ephemeral session key, which is then used to carry on symmetric cryptography for the remainder of the session
• Secure File Transfer Protocol (SFTP): a version of FTP that includes encryption and is used for transferring files between two devices (often a client / server)
• Secure Shell (SSH): remote management protocol, which operates over TCP/IP
• all communications are encrypted
• primarily used by IT administrators to manage devices such as servers and network devices
• Internet Protocol Security (IPSec): an IETF standard suite of protocols that is used to connect nodes (e.g. computers or office locations) together
• IPsec protocol standard provides a common framework for encrypting network traffic and is built into a number of common OSs
• IPsec establishes a secure channel in either transport or tunnel mode
• IPsec performs reauthentication of the client system throughout the connected session in order to detect session hijacking
• IPsec uses two protocols: Authentication Header (AH) and Encapsulating Security Payload (ESP) -- see below
• widely used in virtual private networks (VPNs)
• IPSec provides encryption, authentication and data integrity
• transport mode: only packet payload is encrypted for peer-to-peer communication
• tunnel mode: the entire packet (including header) is encrypted for gateway-to-gateway communication
• security association (SA): represents a simplex communication connection/session, recording any config and status info
• authentication header (AH): provides authentication, integrity, and nonrepudiation; provides assurance of message integrity, authentication and access control, preventing replay attacks; does not provide encryption; like an official authentication stamp, but it's not encrypted so anyone can read it
• encapsulating security payload (ESP): provides encryption of the payload which provides confidentiality and integrity of packet content; works with tunnel or transport mode; provides limited authentication and preventing replay attacks (not to the degree of AH)
• Internet Security Association and Key Management Protocol (ISAKMP): an element of IKE used to organize and manage encryption keys generated/exchanged by OAKLEY and SKEME; a security association is the agreed-upon method of auth and encryption used by two entities; ISAKMP's use of security associations enables IPsec to support multiple simultaneous VPNs from each host; the Oakley protocol specifies a sequence of key exchanges and describes their services (such as identity protection and authentication); and SKEME specifies the actual method of key exchange
• Internet Key Exchange (IKE): a standard protocol used to set up a secure and authenticated communication channel between two parties via a virtual private network (VPN); the protocol ensures security for VPN negotiation, remote host and network access
• WireGuard: a modern, open-source VPN protocol designed to be faster, simpler, and more secure than traditional protocols; uses a lean codebase and operates at the kernel level for high performance
