安全測試與開源風險

Security Testing & Open-Source Risk
01 — 安全測試與開源風險

怎麼在開發中找出漏洞？三種互補的測試：SAST（看原始碼）、DAST（打執行中的應用）、IAST（執行中含內部視角），加上管理第三方相依的 SCA（軟體組成分析）與 API 安全。

🗺️ 今天重點：SAST vs DAST vs IAST → SCA/開源相依風險 → API 安全 → 三者互補。
02 — 三種測試比較
測試視角時機/特點
SAST 靜態看原始碼（白箱）早期即可、找得到根因；易誤報
DAST 動態打執行中應用（黑箱）近真實攻擊、找執行期漏洞；較晚
IAST 互動執行中＋內部插樁結合兩者、較準；需植入探針

💡 記憶：SAST＝讀碼（不執行）、DAST＝打站（不看碼）、IAST＝邊跑邊看內部。三者互補，不互相取代。

03 — 開源相依與 API 風險
SCA：掃描第三方/開源套件的已知漏洞與授權風險（現代應用大半是相依）
SBOM：維護相依清單，漏洞爆發時能快速定位是否受影響
API 安全：驗證授權（避免 BOLA/越權）、速率限制、輸入驗證
04 — 關鍵名詞速查
SAST（靜態應用安全測試）— 分析原始碼找漏洞（白箱、早期）。
DAST（動態應用安全測試）— 攻擊執行中應用（黑箱、近真實）。
IAST（互動式應用安全測試）— 執行中插樁，結合靜動態優點。
SCA（軟體組成分析）— 掃描第三方/開源相依的已知漏洞。
Dependency / Supply Chain（相依/供應鏈）— 引入的套件可能帶漏洞或惡意。
API Security（API 安全）— 授權/越權(BOLA)、速率限制、輸入驗證。
05 — 易混淆釐清

SAST vs DAST：SAST 不執行程式、看原始碼（找得到行號根因、但誤報多）；DAST 不看原始碼、打執行中的應用（接近真實攻擊、但較晚且難定位根因）。

自寫 vs 引入的漏洞：SAST/DAST 找你寫的程式漏洞；SCA 找你引入的第三方套件漏洞——現代應用後者占大宗。

06 — 條文與標準

OWASP 提供 SAST/DAST 工具與 API Security Top 10；NIST SSDF 要求測試與相依管理。Log4Shell 後，SBOM＋SCA 成為相依風險管理標配。

07 — 真實事件：Log4Shell 2021
🔎 一個開源套件，全世界連夜救火

2021 年底 Log4Shell（Log4j 的 Log4j2 漏洞 CVE-2021-44228）爆發：一個被極廣泛使用的 Java 日誌套件存在遠端程式碼執行漏洞，無數應用因『相依的相依』而受影響，卻常不知道自己有用到。全球資安團隊連夜盤點與修補。

教訓：你引入的開源套件就是你的攻擊面。沒有 SBOM/SCA，漏洞爆發時連『有沒有受影響』都答不出來。呼應供應鏈風險（Domain 1）。

08 — CISSP 考點提醒

選最佳答案：『不執行程式、從原始碼找漏洞』→ SAST；『攻擊執行中的應用』→ DAST；『管理開源相依漏洞』→ SCA（＋SBOM）。

常見陷阱：SAST≠DAST（讀碼 vs 打站）；自寫漏洞用 SAST/DAST，引入的相依漏洞用 SCA。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：SAST 讀碼、DAST 打站、IAST 邊跑邊看，三者互補；開源相依用 SCA＋SBOM 管理；Log4Shell 是相依風險的警鐘。

Q1 . SAST 與 DAST 最大差別？
→ SAST 不執行、分析原始碼；DAST 攻擊執行中應用、不看原始碼。

Q2 . 管理第三方/開源套件已知漏洞用什麼？
→ SCA（軟體組成分析），搭配 SBOM。

Q3 . Log4Shell 為何影響如此廣？
→ Log4j 被極廣泛相依，許多應用因『相依的相依』受影響卻不自知。
