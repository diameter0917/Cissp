測試技術輪廓

Testing Techniques Overview
01 — 測試技術輪廓

除了掃描與滲透，還有多種測試技術。今天輪廓化：日誌審查、合成交易、程式碼審查、誤用案例測試、測試涵蓋率、介面測試。重點是『哪個技術驗哪種問題』。

🗺️ 今天重點：log review → 合成交易/RUM → code review（SAST/DAST 預告）→ misuse case → 涵蓋率。
02 — 常見測試技術
技術驗什麼
Log Review 日誌審查事後從日誌找異常/入侵跡象（你本業熟悉）
Synthetic Transaction 合成交易主動模擬使用者交易，監測可用性/效能
Code Review 程式碼審查人工/工具檢視原始碼找漏洞
Misuse Case 誤用案例從攻擊者角度設計負向測試
Interface/API Testing測試系統介面與 API 的輸入處理
03 — 測試涵蓋率與正/負向測試
正向測試：驗『該能用的有用』（正常輸入）
負向/誤用測試：驗『該擋的有擋』（異常/惡意輸入）
Test Coverage：涵蓋的需求/程式比例；高涵蓋≠無漏，但低涵蓋必有盲點
04 — 關鍵名詞速查
Log Review（日誌審查）— 檢視日誌找異常與入侵跡象。
Synthetic Transaction（合成交易）— 主動模擬交易監測服務（vs 被動 RUM）。
RUM（真實使用者監測）— Real User Monitoring，觀測真實流量。
Code Review（程式碼審查）— 靜態檢視原始碼找漏洞（含同儕審查）。
Misuse Case（誤用案例）— 以攻擊者視角設計的負向測試。
Test Coverage（測試涵蓋率）— 測試涵蓋需求/程式碼的比例。
05 — 易混淆釐清

合成交易 vs RUM：合成交易是主動模擬（可在沒人用時就測）；RUM 是被動觀測真實使用者。前者抓問題早、後者反映真實體驗。

正向 vs 負向測試：只做正向（功能正常）會漏掉安全漏洞；誤用案例/負向測試才驗證『擋得住攻擊』。

06 — 條文與標準

NIST SP 800-115 涵蓋技術評估；軟體測試對應 SAST/DAST/IAST（Domain 8 深入）。日誌審查與 SIEM 屬 Domain 7，兩章在『監測』上相通。

07 — 真實事件：從日誌發現入侵
🔎 答案藏在日誌裡

許多入侵是透過日誌審查才被發現——異常登入時間、來源 IP、大量失敗後成功、資料外傳跡象。但前提是日誌有開、有集中、有人看；很多組織日誌淹沒於雜訊或根本沒留存，錯失早期偵測。

教訓：日誌是事後最重要的證據與偵測來源；搭配 SIEM 關聯（Domain 7）才能在海量日誌中抓出訊號。

08 — CISSP 考點提醒

選最佳答案：『沒人用時也要監測交易是否正常』→ 合成交易；『驗證系統擋得住惡意輸入』→ 誤用案例/負向測試；『事後找入侵跡象』→ 日誌審查。

常見陷阱：只做正向測試會漏安全漏洞；高測試涵蓋率不保證沒漏洞，但低涵蓋一定有盲點。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：多種測試技術各驗不同面向：日誌審查找入侵、合成交易主動監測、誤用案例做負向測試、涵蓋率衡量完整度。

Q1 . 主動模擬使用者交易以監測服務的技術？
→ 合成交易（synthetic transaction）。

Q2 . 驗證系統能擋住惡意輸入屬正向還負向測試？
→ 負向/誤用案例測試。

Q3 . 事後追查入侵最重要的資料來源？
→ 日誌（log review，搭配 SIEM）。
