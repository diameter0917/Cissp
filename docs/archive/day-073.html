組態/變更/修補深入

Config/Change/Patch Deep Dive
01 — 中期 D7：維運紀律的深化

今天深入組態基準管理、正式變更管理流程、修補生命週期與弱點優先排序。初期我們知道三者重要,中期要懂『流程怎麼跑、優先序怎麼定』。

🗺️ 今天重點：baseline/CMDB → 變更管理流程(含緊急變更) → 修補生命週期 → 風險導向優先排序。
02 — 組態管理深入
Baseline 基準＋CMDB(組態管理資料庫)記錄已知良好狀態與資產關係
偵測組態漂移:自動比對實際 vs 基準,告警偏離
IaC(基礎設施即程式碼):組態版本控管、可審查、可重現(現代雲端做法)
硬化基準:依 CIS Benchmark / STIG 強化設定
03 — 變更管理流程
申請 → 評估(風險/影響) → CAB 核准 → 測試 → 排程實施 → 驗證 → 回顧
緊急變更(emergency change):可走快速通道,但事後須補審
每個變更要有回退計畫(rollback);高風險變更避開營運尖峰
變更與事件關聯:多數中斷源於未受控變更,變更紀錄是事故根因分析的關鍵
04 — 修補生命週期與優先排序

修補流程:盤點→評估(適用性/風險)→測試→部署→驗證。優先序不只看 CVSS,要綜合 是否對外暴露、是否已被利用(KEV)、資產關鍵性、補償控制有無。對外、可遠端、已有 exploit 的弱點最優先;無法即時修補用補償控制(隔離/虛擬修補/WAF 規則)。

05 — 關鍵名詞速查
Baseline / CMDB（基準/組態庫）— 已知良好狀態與資產關係。
Configuration Drift（組態漂移）— 實際偏離基準,累積風險。
Change Advisory Board（變更顧問委員會）— 評估與核准變更。
Emergency Change（緊急變更）— 快速通道,事後補審。
Patch Lifecycle（修補生命週期）— 盤點→測試→部署→驗證。
CVSS / KEV（評分/已知被利用）— 量化嚴重度;KEV 表示在野利用。
IaC（基礎設施即程式碼）— 可版控/審查/重現的組態。
06 — 易混淆釐清

組態 vs 變更管理:組態管理維持『現在的標準狀態』;變更管理控制『狀態如何被改』。互補。

CVSS 高 ≠ 最該先修:還要看是否對外暴露、是否已被利用(KEV)、資產重要性。內網、不可達的高 CVSS 可能排在對外低 CVSS 之後。

緊急變更不是『免審』:可走快速通道,但事後必須補審與記錄,否則變成繞過控制的後門。

07 — 條文與標準

NIST SP 800-128(組態)/800-40(修補)/800-53 CM 控制族、ITIL 變更管理、CIS Benchmarks / DISA STIG(硬化)、CISA KEV 目錄(優先修補在野利用弱點)。

08 — 真實事件：WannaCry MS17-010
🔎 補丁早有,流程沒跟上

2017 WannaCry 利用 MS17-010(SMB) 漏洞橫掃全球。微軟事前約兩個月已釋出修補,但大量組織因修補流程緩慢、變更顧慮、或仍用過時系統而未部署,釀成大規模災情。這是『有補丁但沒打』的修補管理失敗。

教訓:修補要有流程、依風險排序、處理難修補的舊系統(隔離/補償控制);對外可遠端利用的弱點要走緊急修補通道。掃到(D6)還要修到(D7)才真正降風險。

📌 延伸：修補與穩定的兩難

修補可能引入相容性問題甚至大規模故障(曾有安全更新導致全球當機事件)。故修補要分環試→分批部署(canary/ring)→監控→可回退,在『快速修漏洞』與『避免自我中斷』間平衡。緊急高危漏洞(在野利用)則須加速,接受較高的部署風險。

09 — CISSP 考點提醒

選最佳答案：『防未經核准變更釀事故』→ 變更管理(CAB/回退);『高危無法即時修補』→ 補償控制;『弱點優先序』→ 綜合 CVSS＋暴露＋是否在野利用(KEV)＋資產重要性。

常見陷阱：CVSS 高不必然最先修;緊急變更要事後補審;修補前要測試與可回退;組態(維持) vs 變更(控制改動)。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：組態管理維持基準(CMDB/IaC/防漂移)、變更管理受控(CAB/回退/緊急補審)、修補依風險排序(CVSS+暴露+KEV);WannaCry 是流程失敗。

Q1 . 弱點修補優先序除 CVSS 外還要看什麼？
→ 是否對外暴露、是否已在野被利用(KEV)、資產關鍵性、補償控制有無。

Q2 . 緊急變更可否略過審核？
→ 可走快速通道但事後必須補審與記錄,否則成繞過控制的後門。

Q3 . 組態管理與變更管理的分工？
→ 組態管理維持已知良好基準;變更管理控制狀態如何被改動。
