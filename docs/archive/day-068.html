日誌/合成監控與 BAS

Logging/Synthetic Monitoring & BAS
01 — 中期 D6：監控與攻擊模擬

今天深入日誌審查(你本業)、合成交易/RUM 監控、與入侵與攻擊模擬(BAS)。重點：持續驗證『控制是否仍有效、服務是否正常、攻擊能否被偵測』。

🗺️ 今天重點：日誌審查與完整性 → 合成交易 vs RUM → BAS 入侵攻擊模擬 → 與 SIEM/紅藍隊的關係。
02 — 日誌審查深入
集中與正規化：分散日誌彙整(SIEM)、統一時間(NTP)與格式
完整性保護：write-once、雜湊、權限隔離(防攻擊者刪日誌滅證)
保存期限：依法遵與調查需要(攻擊常潛伏數月，需足夠留存)
關聯分析：單條無意義、跨來源關聯出攻擊鏈
03 — 合成交易 vs RUM
方法原理用途
Synthetic Transaction 合成交易主動模擬使用者操作沒人用時也監測可用性/效能
RUM 真實使用者監測被動觀測真實流量反映真實使用者體驗
04 — BAS 入侵與攻擊模擬

BAS（Breach and Attack Simulation）用自動化工具持續、安全地模擬攻擊技術(對照 MITRE ATT&CK)，驗證偵測與控制是否真的會觸發——介於弱點掃描(找弱點)與紅隊(人工對抗)之間，可頻繁、自動地測『偵測涵蓋率』。

05 — 關鍵名詞速查
Log Review（日誌審查）— 檢視日誌找異常與入侵。
Log Integrity（日誌完整性）— 防竄改/刪除(write-once/雜湊)。
Synthetic Transaction（合成交易）— 主動模擬交易監測服務。
RUM（真實使用者監測）— 被動觀測真實流量。
BAS（入侵與攻擊模擬）— 自動化持續驗證偵測與控制。
MITRE ATT&CK（攻擊技術框架）— TTP 對照，量測偵測涵蓋。
Clipping Level（裁切門檻）— 觸發告警的閾值(過低吵、過高漏)。
06 — 易混淆釐清

合成交易 vs RUM：合成主動(可預先發現問題、覆蓋低流量時段)；RUM 被動(真實體驗、但只反映已發生流量)。

BAS vs 紅隊 vs 弱點掃描：弱點掃描找弱點；BAS 自動模擬攻擊驗偵測(頻繁)；紅隊人工對抗演練(深入、隱蔽)。三者頻率與深度不同。

日誌完整性：攻擊者常刪/改日誌滅證，故日誌須即時外送、防竄改，本機日誌不可全信。

07 — 條文與標準

NIST SP 800-92（日誌管理）、SP 800-137（持續監控 ISCM）、MITRE ATT&CK。合成監控對應 SRE/可用性實務；BAS 屬新興持續驗證工具。

08 — 真實事件：從日誌追入侵
🔎 日誌是事後唯一的真相

重大入侵的調查幾乎都仰賴日誌重建攻擊時間軸：初始入侵點、橫向移動、提權、資料外傳。但常見問題是日誌沒開、留存太短、或被攻擊者刪除，導致無法判定影響範圍與根因。反之，完整且防竄改的日誌讓組織能精準界定外洩範圍(影響法遵通報義務)。

教訓：日誌的覆蓋面、留存期、完整性決定了事後能否查清；搭配 SIEM 關聯(D7)與 UEBA 在事前就偵測。沒有日誌＝盲飛。

📌 延伸：裁切門檻與告警調校

Clipping level（裁切門檻）：設定觸發告警/記錄的閾值(如『5 次失敗登入』)。門檻太低→告警洪水(疲勞)；太高→漏掉慢速攻擊(如密碼噴灑刻意低於門檻)。調校是持續工程——結合 UEBA 的動態基線比靜態門檻更能抓出『低調』攻擊。

09 — CISSP 考點提醒

選最佳答案：『沒人用時也監測服務』→ 合成交易；『真實使用者體驗』→ RUM；『自動持續驗證偵測涵蓋』→ BAS；『事後追入侵範圍』→ 防竄改的日誌。

常見陷阱：本機日誌可能被刪(需即時外送)；clipping level 過高會漏慢速攻擊；合成(主動) vs RUM(被動)。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：日誌要集中/同步/防竄改/夠久；合成交易主動監測、RUM 被動反映體驗；BAS 自動模擬攻擊驗偵測；clipping level 需調校。

Q1 . 為何日誌要即時外送並防竄改？
→ 攻擊者常刪改本機日誌滅證；外送防竄改才保住事後調查真相。

Q2 . BAS 與弱點掃描的差別？
→ 掃描找弱點；BAS 自動模擬攻擊技術驗證偵測與控制是否真會觸發。

Q3 . clipping level 設太高的風險？
→ 漏掉刻意低於門檻的慢速攻擊(如密碼噴灑)。

🎯 實戰演練 EXTRA
QUESTION 1

An online banking service sees almost no traffic between 2 a.m. and 5 a.m., yet management wants assurance that the service is functioning correctly during those hours, before any customer is affected. Which monitoring approach BEST meets this requirement?

（某網路銀行服務凌晨 2 點到 5 點幾乎沒有流量,但管理層希望在任何客戶受影響之前,就能確認服務在該時段運作正常。哪種監控方式『最能』滿足此需求?）

A. Real user monitoring (RUM) 真實使用者監測
B. Daily review of application logs 每日審查應用程式日誌
C. Synthetic transactions executed on a schedule 依排程執行的合成交易
D. A breach and attack simulation (BAS) platform 入侵與攻擊模擬平台
✅ 解答與解析

正解：C。合成交易(synthetic/active monitoring)以腳本主動模擬使用者操作並比對預期結果,即使『沒有真實使用者』也能持續驗證可用性與功能,在客戶受影響前先發現問題——正中題幹『低流量時段』與『事前發現』兩個條件。記憶:沒人用也要測→合成(主動)。

為何其他選項錯：
  • A：RUM 是被動觀測真實流量,凌晨沒人用就沒有資料;等有資料時已有使用者受影響。
  • B：每日日誌審查是事後檢視,無法在凌晨即時發現服務異常。
  • D：BAS 驗證的是『偵測與控制會不會被觸發』,不是服務功能與可用性監控。
QUESTION 2

During an incident investigation, SOC analysts find that firewall, server, and application events cannot be assembled into a coherent timeline because their timestamps disagree by several minutes. Which control would BEST prevent this problem?

（事件調查期間,SOC 分析師發現防火牆,伺服器與應用程式的事件因時間戳相差數分鐘而無法組成一致的時間軸。哪個控制『最能』預防此問題?）

A. Increasing the log retention period 延長日誌保存期限
B. Enabling log normalization on the SIEM 在 SIEM 上啟用日誌正規化
C. Collecting logs from additional sources 蒐集更多來源的日誌
D. Synchronizing all log sources and the SIEM using NTP 以 NTP 同步所有日誌來源與 SIEM 的時鐘
✅ 解答與解析

正解：D。考綱明言:日誌系統應使用 NTP(Network Time Protocol),讓『送出日誌的系統』與『SIEM 本身』時鐘同步,確保多來源資訊有一致的時間軸。時鐘不同步,再多日誌也拼不出攻擊鏈——關聯分析的前提是共同的時間基準。

為何其他選項錯：
  • A：保存期限解決『留多久』,不解決時間戳彼此不一致。
  • B：正規化統一的是格式與欄位,無法修正來源時鐘本身的偏差。
  • C：來源更多但時鐘仍亂,時間軸問題只會更嚴重。
QUESTION 3

Investigators suspect that an attacker moved laterally between internal hosts and exfiltrated a large volume of data to an external address. Which log source would be MOST useful for reconstructing these network communications?

（調查人員懷疑攻擊者曾在內部主機間橫向移動,並將大量資料外傳至外部位址。哪種日誌來源『最有助於』重建這些網路通訊?）

A. Change management logs 變更管理日誌
B. Windows Group Policy Object (GPO) settings Windows 群組原則(GPO)設定
C. Application error logs 應用程式錯誤日誌
D. Network flow (NetFlow) records 網路流量(NetFlow)紀錄
✅ 解答與解析

正解：D。考綱明言:NetFlow 紀錄對調查安全事件特別有用——它記錄『誰跟誰,何時,傳了多少量』的連線中繼資料,正適合重建橫向移動路徑與辨識異常的大量外傳流量。

為何其他選項錯：
  • A：變更管理日誌追蹤系統變更與核准,與網路通訊軌跡無關。
  • B：GPO 是集中部署日誌政策的工具,本身不是通訊紀錄。
  • C：應用程式錯誤日誌只看單一應用的異常,拼不出跨主機的流向與流量。
📑 官方考綱對照重點（Official Objectives · §6.2.3 日誌 / §6.2.9 BAS（深入））
▍SIEM 與日誌

SIEM 能力：aggregation 彙整／normalization／correlation 關聯／安全儲存／分析／報告。日誌須NTP 時間同步、集中唯讀保存防竄改。NetFlow 對調查事件特別有用。

▍BAS 與合成監控

BAS（Breach & Attack Simulation）＝自動化結合 red+blue、模擬 APT、注入威脅指標測控制。參考 MITRE ATT&CK、NIST 800-115、OSSTMM。

§6.2.3 Log reviews
• Security Information and Event Management (SIEM): packages that collect information using the syslog functionality present in many devices, operating systems, and applications
• SIEM capabilities: aggregation, normalization, correlation, secure storage, analysis, reporting
• Log data is recorded in databases; common logs include:
• security, application, firewall, proxy, and change management logs
• Log files should be protected by centrally storing them and using permissions to restrict access, and archived logs should be set to read-only to prevent modification
• Admins may choose to deploy logging policies through Windows Group Policy Objects (GPOs)
• Logging systems should also make use of the Network Time Protocol (NTP) to ensure that clocks are synchronized on systems sending log entries to the SIEM as well as the SIEM itself, ensuring info from multiple sources have a consistent timeline
• Information security managers should also periodically conduct log reviews, particularly for sensitive functions, to ensure that privileged users are not abusing their privileges
• Network flow (NetFlow) logs are particularly useful when investigating security incidents
§6.2.9 Breach attack simulations
• Breach and attack simulation (BAS): platforms that automate some aspects of penetration testing
• The BAS platform helps conduct automated testing of security controls to identify deficiencies
• A BAS system combines red team (attack) and blue team (defense) techniques together with automation to simulate advanced persistent threats (and other advanced threat actors) running against the environment
• Designed to inject threat indicators onto systems and networks in an effort to trigger other security controls (e.g. place a suspicious file on a server)
• detection and prevention controls should immediately detect and/or block this traffic as potentially malicious
• See:
• OWASP Web Security Testing Guide
• OSSTMM (Open Source Security Testing Methodology Manual)
• MITRE ATT&CK framework
• NIST 800-115
• FedRAMP Penetration Test Guidance
• PCI DSS Information Supplemental on Penetration Testing
