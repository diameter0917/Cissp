В§8.5.1 Security weaknesses and vulnerabilities at the source-code level
вҖў A source code vulnerability is a code defect providing a threat actor with an opportunity to compromise the security of a software system
вҖў source code vulns are caused by design or implementation flaws
вҖў design flaw: if dev did everything correctly, there would still be a vulnerability
вҖў implementation flaw: dev incorrectly implemented part of a good design
вҖў the OWASP top 10 vulnerabilities (2025 edition):
вҖў Broken access control
вҖў Security misconfiguration
вҖў Software supply chain failures
вҖў Cryptographic failures
вҖў Injection
вҖў Insecure design
вҖў Authentication failures
вҖў Software or data integrity failures
вҖў Security logging and alerting failures
вҖў Mishandling of exceptional conditions
В§8.5.3 Security coding practices
вҖў Secure coding practices can be summarized as standards and guidelines
вҖў standards: mandatory activities, actions, or rules
вҖў guidelines: recommended actions or ops guidelines that provide flexibility for unforeseen circumstances
вҖў orgs greatly reduce source code vulns by enforcing secure coding standards and maintaining coding guidelines that reflect best practices
вҖў To be considered a standard, coding practice must meet the following:
вҖў reduces the risk of a particular type of vuln
вҖў enforceable across all of an org's software development efforts
вҖў verifiably implemented
вҖў Note: secure coding standards, rigorously applied, is the best way to reduce source code vulns; coding standards ensure devs always do certain things in a certain way, while avoiding others
вҖў Secure coding guidelines are recommended practices that tend to be less specific than standards
вҖў e.g. consistently formatted code comments, or keeping code functions short/tight
