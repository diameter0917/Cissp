考點：測試技術選用

Exam: Test Technique Selection
01 — 考前重點濃縮

本封考測試技術選用:SAST vs DAST、合成交易適用、涵蓋率。判斷情境選哪種測試。

SAST 讀原始碼(早期);DAST 打執行中應用;IAST 插樁;SCA 查相依。
合成交易主動監測(沒人用也測);負向/誤用案例驗安全。
高涵蓋率不保證無漏,但低涵蓋必有盲點。
02 — 核心速記
技術用途
SAST看原始碼找漏洞(早期)
DAST打執行中應用
SCA查第三方相依漏洞
合成交易主動監測可用性/效能
03 — 關鍵名詞（中英）
SAST（靜態應用測試）— 讀原始碼。
DAST（動態應用測試）— 打執行中應用。
SCA（軟體組成分析）— 查相依漏洞。
Synthetic Transaction（合成交易）— 主動模擬監測。
Test Coverage（測試涵蓋率）— 涵蓋需求/程式比例。
04 — 常見陷阱

SAST(讀碼) vs DAST(打站);自寫漏洞 SAST/DAST、引入相依用 SCA。

合成交易主動 vs RUM 被動。

只做正向測試會漏安全漏洞(需負向/誤用)。

05 — 🎯 實戰考題
QUESTION 1

Developers want to find vulnerabilities by analyzing the application's source code WITHOUT running it. Which technique is this?

（開發者想在『不執行程式』下,藉由分析應用原始碼來找漏洞。這是哪種技術?）

A. DAST
B. SAST 靜態應用安全測試
C. Fuzzing
D. Penetration test
✅ 解答與解析

正解：B。SAST 在不執行下分析原始碼找漏洞(早期、白箱),能定位到行號根因。

為何其他選項錯：
  • A：DAST 是執行中攻擊應用(不看原始碼)。
  • C：Fuzzing 餵異常輸入給執行中程式。
  • D：滲透測試針對執行中系統。
QUESTION 2

An organization wants to detect outages of its e-commerce checkout flow even during off-peak hours when few real users are active. Which technique is BEST?

（某組織想偵測其結帳流程的故障,即使在離峰、少有真實使用者時也要偵測。哪種技術最佳?）

A. Real User Monitoring (RUM) 真實使用者監測
B. Synthetic transaction monitoring 合成交易監測
C. SAST
D. Log retention 日誌保存
✅ 解答與解析

正解：B。合成交易主動模擬使用者操作,即使無真實流量時也能持續監測可用性/效能。

為何其他選項錯：
  • A：RUM 被動觀測真實流量,離峰無人時無資料。
  • C：SAST 是程式碼分析,非可用性監測。
  • D：日誌保存非主動偵測故障。
QUESTION 3

Which technique is specifically used to identify known vulnerabilities in third-party and open-source dependencies?

（哪種技術專門用於辨識第三方與開源『相依套件』中的已知漏洞?）

A. SAST
B. DAST
C. SCA 軟體組成分析
D. Synthetic transaction 合成交易
✅ 解答與解析

正解：C。SCA 掃描第三方/開源相依的已知漏洞(配合 SBOM);SAST/DAST 找的是你自己寫的程式漏洞。

為何其他選項錯：
  • A／B：SAST/DAST 針對自寫程式,非相依套件清單。
  • D：合成交易是可用性監測。
06 — (ISC)² 選最佳答案思路

不執行讀碼=SAST;打執行中=DAST;查相依=SCA;主動監測=合成交易。

自寫漏洞 SAST/DAST、引入相依 SCA;離峰也要測可用性→合成交易。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：SAST 讀碼/DAST 打站/SCA 查相依/合成交易主動監測;選技術看『要驗什麼』。

Q1 . 不執行程式、分析原始碼找漏洞是哪種？
→ SAST(靜態)。

Q2 . 離峰也要監測結帳流程可用性用什麼？
→ 合成交易監測。

Q3 . 查第三方相依已知漏洞用什麼？
→ SCA(軟體組成分析)。

§6.2.5 Code review and testing
• Code review and testing is "one of the most critical components of a software testing program"
• These procedures provide third-party reviews of the work performed by developers before moving code into a production environment, possibly discovering security, performance, or reliability flaws in apps before they go live and negatively impact business operations
• In code review, AKA peer review, developers other than the one who wrote the code review it for defects; code review can be a formal or informal validation process
• Fagan inspections: the most formal code review process follows six steps:
• Entry criteria are the criteria or requirements which must be met to enter a specific process
• Exit criteria are the criteria or requirements which must be met to complete a specific process
• Static application security testing (SAST): evaluates the security of software without running it by analyzing either the source code or the compiled application; code reviews are an example of static app security testing
• Dynamic application security testing (DAST): evaluates the security of software in a runtime environment and is often the only option for organizations deploying applications written by someone else
• Interactive Application Security Testing (IAST): combines elements of both SAST and DAST by analyzing code during runtime from within the application, providing more accurate results
