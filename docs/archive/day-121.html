加強：零信任架構深入（NIST SP 800-207）

Deep Dive: Zero Trust Architecture (NIST SP 800-207)
00 — 📬 開場：加強週開跑

歡迎進入加強複習週!接下來 6 封是針對『新考綱重點 + 高頻弱點』的火力加強。今天深入 Zero Trust（零信任）——它是 2024 官方考綱 §3.1 新增的安全設計原則,也是近年情境題的常客。之前 D4 網路篇提過概念,今天用 NIST SP 800-207 把架構元件一次講透。

01 — Never Trust, Always Verify

Zero Trust（零信任）是一種安全模型:不論使用者、裝置或系統位於網路何處,一律不預設信任。每一次存取請求都被視為來自未知、不受信任的位置,必須先驗證身分（authenticate）、授權（authorize）、加密（encrypt）才放行,而且是持續驗證,不是登入一次就永久有效。

Never trust, always verify（永不信任,始終驗證）——取代了舊式的 trust but verify（信任但查證,源自俄國諺語,已被視為不足）。
Assume breach（假設已被入侵）——設計時假設攻擊者已在內網,因此內部流量同樣要驗證與加密。
Least privilege（最小權限）——每次授權只給該請求所需的最小存取,粒度做到最細。
02 — 傳統邊界防禦 vs 零信任（對比表）

傳統模型被稱為 castle-and-moat（城堡與護城河）:邊界防得很硬,但進了城牆就隱性信任——這正是內部威脅與橫向移動（lateral movement）得逞的原因。

面向傳統邊界（castle-and-moat）Zero Trust
信任假設內網＝可信,外網＝不可信任何位置都不預設信任
驗證時機進入邊界時驗證一次每一次請求都持續驗證
安全核心網路位置（IP/網段）身分 + 裝置狀態（identity-centric）
橫向移動入侵後易於橫向擴散微分段 + 逐請求授權,大幅受限
典型技術防火牆、DMZ、VPNZTNA、微分段、MFA、裝置健檢
03 — NIST SP 800-207 核心元件

NIST SP 800-207《Zero Trust Architecture》是零信任的權威參考文件。它把架構拆成控制平面（control plane）與資料平面（data plane）:

元件平面職責
Policy Engine（PE）政策引擎控制大腦:依政策、身分、裝置狀態、威脅情資做出准駁決策（grant/deny/revoke）
Policy Administrator（PA）政策管理器控制傳令官:把 PE 的決策下達給 PEP,建立/切斷通訊路徑（發 token/憑證）
Policy Enforcement Point（PEP）政策執行點資料守門員:位於主體與資源之間,實際執行放行或阻斷每個連線

PE + PA 合稱 Policy Decision Point（PDP,政策決策點）。記憶法:PE 決策、PA 下令、PEP 執行——「法官、書記官、法警」。

Subject 主體 → PEP → Resource 資源
↑ 控制平面: PDP = PE + PA 對 PEP 下達准駁指令
04 — ZTNA vs VPN

ZTNA（Zero Trust Network Access,零信任網路存取）常被稱為「VPN 的繼任者」,也是 SASE 框架的關鍵元件之一。差異在於:

面向傳統 VPNZTNA
存取範圍通了隧道 = 進入整個內網（過度授權）只連到特定應用,逐應用授權
驗證連線時驗證一次持續評估身分 + 裝置狀態（posture）
暴露面VPN 閘道對外公開,常成攻擊目標應用對外隱形（dark cloud）,由 broker 中介
橫向移動風險高:憑證失竊即可漫遊內網低:看不到未授權的資源
05 — 微分段（Microsegmentation）

Microsegmentation（微分段）是零信任在網路層的落地手段:把網路切成非常小的安全區段——細到單一工作負載（workload）、單一 VM 或容器——每個區段之間的東西向流量（east-west traffic）都要通過政策檢查。與傳統 VLAN 分段的差別:VLAN 以「網段」為單位、段內互信;微分段以「工作負載」為單位、段內也不互信。它是限制橫向移動、圍堵勒索軟體擴散的首選答案,常以軟體定義網路（SDN）或主機防火牆政策實作。

06 — 零信任七大支柱（精簡版）

美國 CISA 的 Zero Trust Maturity Model 把零信任拆成五大支柱 + 兩個橫貫能力,合稱七大面向,是盤點成熟度的實用框架:

① Identity 身分 — MFA、持續驗證、以身分為新邊界。
② Devices 裝置 — 裝置清冊、健康狀態（posture）檢查後才給存取。
③ Networks 網路 — 微分段、全程加密、不再信任「內網」。
④ Applications & Workloads 應用與工作負載 — 逐應用授權、安全開發與部署。
⑤ Data 資料 — 分類分級、加密、DLP——零信任的最終保護對象。
＋ Visibility & Analytics／Automation & Orchestration — 橫貫全部支柱的可見性分析與自動化編排。
07 — 實務案例：Google BeyondCorp

2009 年 Google 遭遇國家級攻擊 Operation Aurora（極光行動),攻擊者突破邊界後在內網橫向移動。Google 痛定思痛,啟動 BeyondCorp 計畫:徹底廢除「特權內網」,員工不論在辦公室或咖啡廳,一律走相同的存取流程——由 access proxy（相當於 PEP）依「使用者身分 + 裝置清冊與健康狀態」逐請求授權,內網 IP 不再代表任何信任。

啟示:BeyondCorp 是零信任「去邊界化」最著名的實證——不靠 VPN,靠身分與裝置狀態決定一切存取。

08 — 🚨 CISSP 考點雷達

① 考綱定位:零信任是 2024 考綱 §3.1 新增的安全設計原則（3.1.8 Zero Trust or trust but verify）,屬於「必然出題」的新重點。

② 情境題常問「哪個做法最能落實零信任」:正解特徵是「持續驗證每一請求、以身分/裝置狀態為依據、最小權限」;干擾項特徵是「加強邊界」（更大防火牆、更多 VPN）——那是 castle-and-moat 思維。

③ 元件題:分清 PE 決策、PA 下令、PEP 執行;PDP = PE + PA。限制橫向移動 → 想到微分段。

🎯 實戰演練
QUESTION 1

A CISO is redesigning security architecture after attackers breached the perimeter and moved laterally for months. Which approach BEST implements a zero trust design principle?

（攻擊者突破邊界後在內網橫向移動數月,CISO 要重新設計安全架構。哪個做法『最能』落實零信任設計原則?）

A. Continuously verify every access request based on identity and device posture, regardless of network location 不論網路位置,依身分與裝置狀態持續驗證每一次存取請求
B. Deploy a next-generation firewall at the network perimeter 在網路邊界部署次世代防火牆
C. Require all remote users to connect through a full-tunnel VPN 要求所有遠端使用者一律走 full-tunnel VPN
D. Authenticate users once at login and trust the session all day 登入時驗證一次,整天信任該工作階段
✅ 解答與解析

正解：A。零信任核心是 never trust, always verify:不以網路位置為信任依據,對每一次請求依身分與裝置狀態持續驗證授權——這正是防止橫向移動的根本解。記憶法:題目看到『橫向移動 + 最能落實零信任』,選『持續驗證/身分中心』,不選『加強邊界』。

為何其他選項錯：
  • B：更強的邊界防火牆仍是 castle-and-moat 思維,擋不住已入內網的橫向移動。
  • C：VPN 通了隧道即給整個內網存取,屬過度授權,與零信任相悖。
  • D：驗證一次就長期信任,違反『持續驗證』原則。
QUESTION 2

An architect is deploying zero trust per NIST SP 800-207. Which component sits in the data path and actually enables, monitors, and terminates each connection between a subject and a resource?

（依 NIST SP 800-207 部署零信任,哪個元件位於資料路徑上,實際負責放行、監控與終止主體到資源的每一條連線?）

A. Policy Engine 政策引擎
B. Policy Administrator 政策管理器
C. Policy Enforcement Point 政策執行點
D. Policy Decision Point 政策決策點
✅ 解答與解析

正解：C。PEP（Policy Enforcement Point）是資料平面上的守門員,位於主體與資源之間,依控制平面的指令實際執行放行/監控/切斷連線。記憶法:PE 決策、PA 下令、PEP 執行（法官、書記官、法警）。

為何其他選項錯：
  • A：Policy Engine 在控制平面『做決策』,不在資料路徑上執行。
  • B：Policy Administrator 負責把決策『下達』給 PEP,本身不擋流量。
  • D：PDP = PE + PA 的合稱,屬控制平面,不是執行元件。
QUESTION 3

After a ransomware incident spread from one compromised server to dozens of others in the same data center, which control would BEST limit lateral movement in line with zero trust?

（勒索軟體從一台受駭伺服器擴散到同機房數十台主機。哪個控制『最能』依零信任精神限制橫向移動?）

A. Deploy a network intrusion detection system (NIDS) 部署網路入侵偵測系統
B. Move the servers into a hardened DMZ 把伺服器搬進強化的 DMZ
C. Separate servers into a few large VLANs 將伺服器劃分成幾個大型 VLAN
D. Implement microsegmentation with per-workload access policies 實施微分段,對每個工作負載套用個別存取政策
✅ 解答與解析

正解：D。Microsegmentation（微分段）把安全邊界縮到單一工作負載,東西向流量逐一過政策檢查——即使一台被攻陷,也無法自由連到鄰居,是圍堵橫向移動的最佳答案。判斷邏輯:『限制 lateral movement』幾乎等於在點名微分段。

為何其他選項錯：
  • A：NIDS 只能『偵測』,不能阻斷擴散（偵測性 vs 預防性控制）。
  • B：DMZ 是對外服務的邊界概念,無助於內部東西向流量管控。
  • C：大型 VLAN 段內仍互信,勒索軟體照樣在段內橫掃,粒度不足。
✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：零信任 = never trust, always verify:不看網路位置、看身分與裝置狀態,每次請求都驗證;SP 800-207 由 PE 決策、PA 下令、PEP 執行;微分段擋橫向移動。

Q1 . PDP 由哪兩個元件組成？
→ Policy Engine + Policy Administrator（PEP 是資料平面的執行點）。

Q2 . ZTNA 與 VPN 最大的差異？
→ ZTNA 逐應用授權、持續驗證、應用對外隱形;VPN 通了隧道即給整個內網。

Q3 . Google BeyondCorp 的轉型契機與核心做法？
→ 2009 Operation Aurora 後廢除特權內網,由 access proxy 依身分 + 裝置狀態逐請求授權。

📑 官方考綱對照重點（Official Objectives · §3.1 零信任設計原則）
▍§3.1.8 Zero Trust or trust but verify（2024 新增）

Zero Trust:假設任何使用者/裝置/系統都不預設可信（assume breach）,對每個請求強制持續驗證、最小權限、細粒度存取控制;取代傳統 castle/moat 與 trust but verify。目標是每次存取都先 authenticated、authorized、encrypted 才放行。

▍架構參考:NIST SP 800-207

控制平面 PDP（Policy Engine + Policy Administrator）做決策並下令,資料平面 PEP 執行放行/阻斷;搭配 §3.1.11 SASE 的 ZTNA 元件與微分段,實現 identity-centric 的去邊界架構。
