DLP/DRM/CASB 與隱私設計

DLP/DRM/CASB & Privacy by Design
01 — 深入資料保護技術與隱私設計

D2 中期收尾，深入三大技術的運作模式與適用（DLP/DRM/CASB），並把 Privacy by Design（隱私設計）七原則與去識別化技術（匿名/假名/代碼化）講透——這是把『保護資料』從工具層提升到設計層。

🗺️ 本週回顧：①分類制度 → ②角色責任 → ③生命週期/主權 → ④殘留/銷毀 → ⑤今天：DLP/DRM/CASB 與 PbD。下週進入 Domain 3 中期。
02 — DLP 的三種部署
DLP 類型位置攔什麼
Network DLP網路出口外送的郵件/上傳（in transit）
Endpoint DLP端點USB 複製、列印、剪貼（in use）
Storage/Discovery DLP儲存庫掃描靜止資料找未受保護的敏感檔（at rest）

DLP 依分類標示與內容比對（regex/指紋/機器學習）判斷敏感與否——分類做得好，DLP 才準（呼應 Day 46）。三種部署正好對應資料三狀態。

03 — CASB 四支柱與 DRM/IRM
CASB 四支柱：可視化（Visibility，含影子 IT 發現）、合規（Compliance）、資料安全（DLP/加密）、威脅防護（Threat Protection）
DRM/IRM：把保護綁在檔案本身（加密＋使用權限），檔案離開組織仍受控（可限開啟/複製/列印/期限）
差異：DLP 在出口攔截；DRM 讓檔案到哪都受控；CASB 管雲端的整體可視與管控
04 — Privacy by Design 與去識別化

Privacy by Design（PbD）七原則核心：主動非被動、預設即隱私（by default）、內建到設計、正和（不犧牲功能換隱私）、全生命週期保護、透明、尊重使用者。去識別化技術：

技術作法可逆?
Anonymization 匿名化永久移除可識別性不可逆
Pseudonymization 假名化以代號替換、金鑰可還原可逆(持金鑰)
Tokenization 代碼化以無意義 token 替換敏感值可逆(查表)
05 — 關鍵名詞速查
DLP（資料外洩防護）— 網路/端點/儲存三種部署，依分類攔截。
CASB（雲端存取安全代理）— 可視化/合規/資料安全/威脅防護四支柱。
DRM/IRM（數位/資訊版權管理）— 保護綁在檔案上，離開組織仍受控。
Privacy by Design（隱私設計）— 設計階段即內建隱私、預設最隱私。
Anonymization（匿名化）— 永久不可逆地移除可識別性。
Pseudonymization（假名化）— 可逆替換，GDPR 認可的降風險措施。
Tokenization（代碼化）— 以 token 替敏感值（如付款）。
06 — 易混淆釐清

DLP vs DRM：DLP 著重『阻止外流』（出口把關）；DRM 著重『檔案到哪都受控』（持續權限）。外寄後仍要管＝DRM；防它寄出去＝DLP。

Anonymization vs Pseudonymization：匿名化不可逆、脫離個資範疇；假名化可逆（持金鑰可還原）、仍屬個資受 GDPR 規範。考試常考此差異。

Tokenization vs Encryption：加密以演算法＋金鑰轉換（數學可逆）；代碼化以查表替換（token 與原值無數學關係），常用於降 PCI 範圍。

07 — 條文與標準

GDPR Art.25（Privacy by Design & by Default）、Art.4(5)（假名化定義）；ISO/IEC 27701（PIMS）、ISO/IEC 29100（隱私框架）。PCI DSS 鼓勵代碼化以縮小持卡資料範圍。PbD 由 Ann Cavoukian 提出、被 GDPR 法制化。

08 — 真實事件：PbD 落實 vs 事後補救
🔎 把隱私內建，而非事後道歉

許多產品在出事後才『加上隱私設定』，預設卻是最大蒐集、最公開，引發監管與信任危機。反例是落實 Privacy by Default 的服務：新帳號預設最小蒐集、最嚴privacy 設定，使用者要主動放寬才分享更多。GDPR Art.25 正是把這從『好意』變成『法律義務』。

教訓：隱私若是設計階段內建（左移），成本低且建立信任；若事後補，代價是罰款、重構與商譽。呼應 D8 的安全左移。

📌 延伸：假名化在分析場景

企業想用客戶資料做分析又要降風險，常用假名化：把姓名/ID 換成代號、金鑰另存隔離，分析團隊只見假名資料。但要注意：假名化仍是個資（可還原），且若分析結果＋其他資料可重新識別（re-identification），風險依舊。真正脫離 GDPR 需匿名化（不可逆且無法重識別）。

09 — CISSP 考點提醒

選最佳答案：『防敏感檔外傳』→ DLP；『檔案外寄後仍限制開啟/列印』→ DRM；『管雲端 SaaS 與影子 IT』→ CASB；『可逆、仍屬個資的降風險替換』→ 假名化；『不可逆、脫離個資』→ 匿名化。

常見陷阱：匿名(不可逆/非個資) vs 假名(可逆/仍個資) 別反；PbD 的精髓是『預設即隱私』與『內建非事後』。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：DLP(網路/端點/儲存)擋外流、DRM 讓檔案到哪都受控、CASB 管雲端四支柱；PbD 預設即隱私、內建非事後；匿名不可逆/假名可逆仍個資。

Q1 . 匿名化與假名化最大差異？
→ 匿名不可逆、脫離個資；假名可逆(持金鑰)、仍屬個資受 GDPR。

Q2 . 要讓檔案外寄後仍限制開啟與列印，用什麼？
→ DRM/IRM（保護綁在檔案上）。

Q3 . Privacy by Design 的核心預設原則是什麼？
→ Privacy by Default——預設即最小蒐集、最嚴隱私設定。

🎯 實戰演練 EXTRA
QUESTION 1

An investigation reveals that employees in a design department are copying confidential drawings to personal USB drives and printing them at office printers. Which DLP deployment BEST addresses this specific leakage channel?

（調查發現設計部門員工把機密圖檔複製到私人 USB 隨身碟,並在辦公室印表機列印。哪種 DLP 部署最能處理這種特定外洩管道?）

A. Network DLP at the internet egress 部署於網際網路出口的 Network DLP
B. Storage/discovery DLP scanning file repositories 掃描檔案儲存庫的 Storage/Discovery DLP
C. Endpoint DLP agents on the department's workstations 在該部門工作站安裝 Endpoint DLP 代理程式
D. A CASB monitoring sanctioned SaaS applications 監控已核准 SaaS 應用的 CASB
✅ 解答與解析

正解：C。USB 複製與列印發生在端點,屬於資料使用中(in use)的狀態,只有 Endpoint DLP 能在裝置上攔截複製到隨身碟,列印,剪貼等動作。記憶:三種 DLP 對應資料三狀態——網路(in transit),端點(in use),儲存(at rest);先辨識外洩管道在哪一層,再選部署。

為何其他選項錯：
  • A：Network DLP 只看得到『經網路外送』的流量;USB 複製與本機列印根本不經過網路出口。
  • B：Storage DLP 是掃描靜止資料,找出未受保護的敏感檔,無法即時攔截端點上的動作。
  • D：CASB 管的是雲端服務的可視化與管控,與 USB/列印這類本機管道無關。
QUESTION 2

A CISO discovers through expense reports that business units have subscribed to dozens of unsanctioned SaaS applications where corporate files are being shared. Which solution BEST provides visibility into this shadow IT and enforces the organization's cloud usage policy?

（CISO 從報帳紀錄發現各事業單位自行訂閱了數十個未核准的 SaaS,公司檔案正在其中流通。哪個方案最能提供影子 IT 的可視性,並執行組織的雲端使用政策?）

A. Endpoint DLP to block file copies to USB media 以 Endpoint DLP 阻擋檔案複製到 USB
B. DRM to attach usage rights to every corporate document 以 DRM 為每份公司文件附加使用權限
C. A firewall rule blocking all cloud storage domains 以防火牆規則封鎖所有雲端儲存網域
D. A CASB positioned between users and cloud services 部署介於使用者與雲端服務之間的 CASB
✅ 解答與解析

正解：D。CASB 邏輯上位於使用者與雲端服務之間,四支柱的第一項就是可視化(含影子 IT 發現),再配合合規,資料安全(DLP/加密),威脅防護,對雲端使用執行政策——正是為『發現並治理未核准 SaaS』而生的方案。

為何其他選項錯：
  • A：Endpoint DLP 管 USB/列印等端點管道,無法盤點與治理雲端 SaaS 的整體使用。
  • B：DRM 保護單一檔案本身,但不能發現有哪些影子 IT,也無法管控 SaaS 使用行為。
  • C：全面封鎖過於粗暴會癱瘓合法業務,使用者將改走個人裝置繞過,可視性反而更差。
QUESTION 3

A retailer wants to shrink its PCI DSS assessment scope. It plans to replace stored card numbers with surrogate values that have no mathematical relationship to the originals and can only be mapped back through a secured lookup vault. Which technique does this describe?

（零售商想縮小 PCI DSS 評估範圍,計畫把儲存的卡號換成與原值『無數學關係』的替代值,且只能透過受保護的對照庫(vault)換回。這描述的是哪種技術?）

A. Format-preserving encryption 保留格式加密
B. Tokenization 代碼化
C. One-way hashing 單向雜湊
D. Anonymization 匿名化
✅ 解答與解析

正解：B。Tokenization(代碼化)的招牌特徵:token 與原值無數學關係,靠查表(vault)還原。持卡資料以 token 取代後,多數系統不再接觸真卡號,有效縮小 PCI 範圍——這正是 PCI DSS 鼓勵代碼化的原因。判斷關鍵:數學轉換=加密;查表替換=代碼化。

為何其他選項錯：
  • A：加密(含保留格式加密)是演算法+金鑰的數學轉換,密文與原值存在數學關係,與題幹矛盾。
  • C：單向雜湊不可還原,無法透過對照庫換回原卡號,不符『可換回』的需求。
  • D：匿名化是永久不可逆地移除可識別性,同樣無法還原,且是個資去識別概念而非替代值機制。

▍保護技術細節

DLP 三型：network（網路邊緣掃外送）／endpoint（端點，可擋列印/複製到隨身碟）／cloud。CASB：邏輯上位於使用者與雲端之間，四基石＝可視性/資料安全/威脅偵測/合規。DRM：隨資料移動，適合無法加密磁碟區時。

▍隱私去識別化

Anonymization 匿名化（永久不可逆）vs Pseudonymization 假名化／Tokenization 代碼化／加密（有金鑰/對照表可逆）。

對應官方 §2.6.4 DRM/DLP/CASB 與去識別化。

§2.6.4 Data protection methods (e.g., Digital Rights Management (DRM), Data Loss Prevention (DLP), Cloud Access Security Broker (CASB))
• Data protection methods include:
• digital rights management (DRM): methods used in attempt to protect copyrighted materials; purpose is to prevent the unauthorized use, modification, and distribution of copyrighted works
• Data Loss Prevention (DLP): systems that detect and block data exfiltration attempts by monitoring data in motion, at rest, and in use; three primary types:
• network-based DLP: placed on the edge of a network, scans all outgoing network data
• endpoint-based DLP: scans stored files and can be used to prevent printing or copying sensitive data to removable storage
• cloud-based DLP: similar to network DLP but designed specifically for cloud-native environments
• Cloud Access Security Brokers (CASBs): software placed logically between users and cloud-based resources ensuring that cloud resources have the same protections as resources within a network
• CASB is a solution for security policy enforcement, ensuring security policies and compliance are met when accessing cloud apps and data; it can be used on-premise or in the cloud
• the four cornerstones of CASBs are visibility, data security, threat detection, and compliance
• note that entities must comply with the EU GDPR, and use additional data protection methods/controls such as anonymization and randomized masking (which, when done correctly, can't be reversed), or quasi-anonymization which can be reversed (pseudonymization, tokenization, encryption)
• One of the primary methods of protecting the confidentiality of data is encryption
• Options for protecting your data vary depending on its state:
• Data at rest: consider encryption for operating system volumes and data volumes, and backups as well
• be sure to consider all locations for data at rest, such as tapes, USB drives, external drives, RAID arrays, SAN, NAS, and optical media
• DRM is useful for data at rest because DRM "travels with the data" regardless of the data state
• DRM is especially useful when you can’t encrypt data volumes
• Data in transit: think of data in transit wholistically -- moving data from anywhere to anywhere; use encryption for data in transit
• e.g. a web server uses a certificate to encrypt data being viewed by a user, or IPsec encrypting a communication session
• most important point is to use encryption whenever possible, including for internal-only web apps
• DLP solutions are useful for data in transit, scanning data on the wire, and stopping the transmission/transfer, based on the DLP rules set (e.g. outbound data that contains numbers matching a social security number pattern, a DLP rule can be used to block that traffic)
• Data in use:
• CASB solution often combines DLP, a web application firewall with some type of authentication and authorization, and a network firewall in a single solution; A CASB solution is helpful for protecting data in use (and data in transit)
• Pseudonymization: refers to the process of using pseudonyms to represent other data; process of replacing data elements with pseudonyms or aliases
• A pseudonym is an alias, and pseudonymization can prevent data from directly identifying an entity (i.e. person)
• An external dataset holds the original data along with the pseudonym such that the original data can be recreated
• Therefore, unlike full anonymization, pseudonymized data can be reversed if the key is available
• Tokenization: use of a token, typically a random string of characters, to replace other data
• note that tokenization is similar to pseudonymization in that they are both used to represent other data, and the token or pseudonym have no meaning or value outside the process that creates and links them to that data
• example of tokenization used in CC transactions:
• registration: app on user's smart phone securely sends CC info to the credit card processor (CCP)
• The CCP sends the CC info to a tokenization vault, creating a token and associating it with the user's phone
• usage: when the user makes a purchase, the POS system sends the token to the CCP for authorization
• validation: the CCP sends the token to the tokenization vault; the vault replies with the CC info, the charge is processed
• completing the sale: the CCP sends a reply to the POS indicating the charge is approved
• this system prevents CC theft at the POS system
