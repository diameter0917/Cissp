外購軟體與惡意程式

Acquired Software & Malware
01 — 外購軟體與惡意程式

Domain 8 與整個初期收尾：不是所有軟體都自己寫——COTS、開源、託管（SaaS）各有風險；還要認識惡意程式類型與上線驗收。核心：引入的軟體也要評估與驗收。

🗺️ 初期收尾：D1–D8 全貌已建立。下週進入中期 P2，回到 D1 逐一深入名詞、條文與案例。
02 — 外購軟體風險
來源風險重點
COTS 商用現成黑箱、漏洞依賴原廠修補、授權合規
Open Source 開源相依漏洞/惡意套件、維護中斷、授權義務
Managed/SaaS 託管資料主權、共同責任、退出策略(lock-in)
03 — 惡意程式類型
Virus/Worm：病毒需宿主、蠕蟲自我傳播
Trojan：偽裝正常程式；Ransomware：加密勒索
Rootkit：隱藏自身、取得高權限；Spyware/Keylogger：竊資
Logic Bomb：條件觸發；Fileless：無檔案、藏記憶體
04 — 關鍵名詞速查
COTS（商用現成軟體）— Commercial Off-The-Shelf，黑箱、靠原廠修補。
Software Acceptance（軟體驗收）— 上線前驗證安全與功能符合需求。
Worm（蠕蟲）— 自我複製傳播、不需宿主。
Trojan（木馬）— 偽裝成正常程式的惡意軟體。
Rootkit（隱匿程式）— 隱藏自身並取得高權限。
Logic Bomb（邏輯炸彈）— 滿足條件才觸發的惡意碼。
05 — 易混淆釐清

Virus vs Worm：病毒需要宿主檔案且常需使用者執行；蠕蟲自我傳播、無需宿主（如 WannaCry、Mirai）。

Trojan vs Virus：木馬偽裝誘騙安裝、不自我複製；病毒會感染其他檔案。

06 — 條文與標準

外購/第三方軟體納入供應鏈風險管理（C-SCRM, NIST SP 800-161）；上線前須驗收測試＋安全評估；開源須遵授權並用 SCA 管漏洞。EDR/防毒屬偵測層（Domain 7）。

07 — 真實事件：供應鏈植入惡意碼
🔎 信任的軟體，藏著後門

攻擊者把惡意碼植入受信任的軟體更新或套件（如 SolarWinds 2020、惡意 npm/PyPI 套件），使用者正常安裝官方更新卻引入後門。因為來源『可信』，傳統防護難以察覺，影響面極廣。

教訓：外購/引入的軟體不是『買來就安全』——需供應商評估、完整性驗證（簽章/雜湊）、SBOM、上線驗收，並持續監控。收束整個 Domain 8 與供應鏈主題。

08 — CISSP 考點提醒

選最佳答案：『自我傳播不需宿主』→ Worm；『偽裝誘騙安裝』→ Trojan；『條件觸發』→ Logic Bomb；『外購軟體上線前』→ 安全驗收＋完整性驗證。

常見陷阱：病毒(需宿主) vs 蠕蟲(自傳播) 別搞混；COTS/開源/SaaS 各有不同風險面。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：外購軟體（COTS/開源/SaaS）各有風險、須評估與驗收；認識惡意程式類型；供應鏈植入是高隱蔽威脅，靠完整性驗證＋SBOM＋監控。

Q1 . 蠕蟲與病毒最大差別？
→ 蠕蟲自我傳播、不需宿主；病毒需宿主檔案。

Q2 . 引入外購/開源軟體前後的關鍵控制？
→ 供應商評估、完整性驗證(簽章/雜湊)、SBOM/SCA、上線安全驗收。

Q3 . 偽裝成正常程式誘騙安裝的惡意軟體？
→ Trojan 木馬（不自我複製）。

🎯 實戰演練 EXTRA
QUESTION 1

A database administrator secretly inserts code into a production system that will erase critical tables if his employee ID ever disappears from the payroll file. Which type of malware does this BEST describe?

（某資料庫管理員在正式系統中偷偷植入程式碼,一旦他的員工編號從薪資檔中消失,就會刪除關鍵資料表。這最符合哪種惡意程式?）

A. Worm 蠕蟲
B. Trojan 木馬
C. Rootkit 隱匿程式
D. Logic bomb 邏輯炸彈
✅ 解答與解析

正解：D。Logic bomb 的識別特徵＝『滿足特定條件才觸發』的惡意碼,典型情境正是內部人員預埋報復程式（被解僱/名字從薪資檔消失時發作）。判斷邏輯:看到 if/當某條件成立才發作→邏輯炸彈;這也是內部威脅（insider threat）常考的組合。

為何其他選項錯：
  • A：Worm 的特徵是自我複製並透過網路自動傳播;此程式碼不傳播、只等條件觸發。
  • B：Trojan 靠偽裝成正常程式誘騙使用者安裝;此碼由內部人直接植入,沒有偽裝誘騙環節。
  • C：Rootkit 的重點是隱藏自身存在並維持高權限,不是條件觸發的破壞行為。
QUESTION 2

A security manager must evaluate a niche COTS application before purchase. The product has little public security research, and the vendor will not release source code. Which of the following BEST helps assess the security of this product?

（安全經理須在採購前評估一套小眾 COTS 應用。該產品的公開安全研究極少,且供應商不提供原始碼。下列何者最有助於評估其安全性?）

A. Research the vendor's security certifications (e.g., ISO/IEC 27034, ISO/IEC 27001) and its vulnerability-handling history 查證供應商安全認證（如 ISO/IEC 27034、ISO/IEC 27001）與其漏洞處理紀錄
B. Require the vendor to hand over source code for in-house SAST 要求供應商交出原始碼供內部執行 SAST
C. Assume it is secure because widely used COTS products are tested by many security researchers 認定其安全,因為廣泛使用的 COTS 已被大量安全研究者測試
D. Add a contract clause requiring external audits of the development process 在合約中加入開發過程外部稽核條款
✅ 解答與解析

正解：A。官方考綱指出:對小眾（niche）COTS 產品,應查證供應商認證（如 ISO/IEC 27034 應用安全、ISO/IEC 27001 ISMS、FedRAMP）並研究其已揭露漏洞與修補紀錄,以判斷供應商對安全的重視程度。判斷邏輯:COTS 是黑箱＋採購前評估→評的是『供應商』的安全成熟度,不是產品內部。

為何其他選項錯：
  • B：COTS 是不供修改的成品,供應商通常不會提供原始碼,此路不可行。
  • C：『廣泛使用才有大量研究者檢視』的前提在小眾產品不成立,正是題幹刻意排除的情境。
  • D：合約綁外部稽核適用於『第三方客製開發』;現成 COTS 已開發完成,沒有開發過程可供稽核。
📑 官方考綱對照重點（Official Objectives · §8.4 外購軟體）
▍外購軟體類型

COTS 商用現成（不改、多經研究者測試，查 ISO/IEC 27034）／Open source 開源（可審程式但無支援、怕過時）／Third-party 客製（須合約綁外部稽核）／Managed services／Cloud（SaaS/IaaS/PaaS）。

▍評估重點

Defensive programming 防禦式編程（視所有輸入為不可信）。雲端外洩主因＝錯誤設定、存取控制不良。SBOM 管依賴。

📂 完整考綱原文節錄（點開看官方逐條，English 原文）

對應官方 §8.4 外購軟體安全。

§8.4.1 Commercial-off-the-shelf (COTS)
• Commercial Off-the-Shelf (COTS): software elements, usually apps, that are provided as finished products (not intended for alteration by or for the end-user)
• Most widely used COTS software products have been tested by security researchers (both benign and malicious)
• researching discovered vulnerabilities and exploits can help us understand how seriously the vendor takes security
• for niche products, you should research vendor certifications, such as ISO/IEC 27034 Application Security
• other than secure coding certification, you can look for overall information security management system (ISMS) certifications such as ISO/IEC 27001 and FedRAMP (which are difficult to obtain, and show that the vendor is serious about security)
• If you can talk with a vendor, look for processes like defensive programming, which is a software development best practice that means as code is developed or reviewed, they are constantly looking for opportunities for things to go badly
• e.g. treating all input routines as untrusted until proven otherwise
§8.4.2 Open source
• Open source is typically released with licensing allowing code access and inspection so devs can look for security issues
• typically, however, this means that there is no service or support that comes with the software and requires in-house support for configuration, and security testing
• it also means that both open-source devs as well as adversaries are able to review the code for vulns
• the greatest risk of open-source software is relying on outdated versions -- especially true of shared libraries
• an org should develop processes to ensure that all open-source software is periodically updated, likely in a way that differs from the process for updating COTS
