密碼分析攻擊與 IoT/ICS

Cryptanalytic Attacks & IoT/ICS
01 — 中期 D3：攻擊密碼學與嵌入式系統

今天兩個主題：密碼分析攻擊（cryptanalytic attacks）——攻擊者怎麼破密；以及 IoT/ICS（工控）嵌入式系統的特殊風險。理解攻擊面，才知道為何前幾天的選擇（金鑰長度、PFS、慢雜湊）如此重要。

🗺️ 今天重點：暴力/字典/生日攻擊 → 旁通道 → 已知/選擇明文 → IoT/ICS 風險與 Stuxnet。
02 — 常見密碼分析攻擊
攻擊原理對策
Brute Force 暴力試遍所有金鑰夠長的金鑰、限速/鎖定
Dictionary/Rainbow常見密碼/預算雜湊表加鹽、慢雜湊
Birthday 生日攻擊利用碰撞機率找雜湊碰撞夠長雜湊(SHA-256+)
Side-channel 旁通道測時間/功耗/電磁推金鑰常數時間實作、屏蔽
Known/Chosen Plaintext用已知/選擇明文密文對強演算法、隨機化(IV)
03 — 旁通道與實作弱點

旁通道攻擊不打數學、而打實作：量測加密運算的時間、功耗、電磁輻射、甚至聲音來推金鑰（如 timing attack、Spectre/Meltdown 屬 CPU 旁通道）。對策是常數時間演算法、加入雜訊/屏蔽。再強的演算法，實作不當仍會洩密。

04 — IoT/ICS 嵌入式風險
特性：算力/記憶體有限、難更新、生命週期長、預設密碼、實體可接觸
ICS/SCADA：工控系統重可用性與安全(safety)，常用老舊協定、不易停機修補
對策：網路分段（IT/OT 隔離）、最小化對外、安全開機、韌體簽章、預設密碼管理
IT vs OT 優先序：IT 重 CIA(機密優先)，OT 重 AIC（可用/安全優先）
05 — 關鍵名詞速查
Brute Force（暴力攻擊）— 窮舉金鑰；長金鑰可抵抗。
Birthday Attack（生日攻擊）— 利用碰撞機率攻擊雜湊。
Side-channel（旁通道）— 由時間/功耗/電磁推測密鑰。
Ciphertext-only（唯密文攻擊）— 只握有密文樣本,靠大量密文分析還原明文或金鑰,最常見也最難成功。
Known-plaintext（已知明文攻擊）— 同時握有明文與對應密文,據以推導金鑰;linear cryptanalysis 屬之。
Chosen-plaintext（選擇明文攻擊）— 可自選任意明文取得對應密文來分析,比已知明文更強;differential cryptanalysis 屬之。
Chosen-ciphertext（選擇密文攻擊）— 可自選密文取得解密結果反推金鑰,比選擇明文更強;對策是 GCM/CCM 認證加密。
Meet-in-the-middle（中間相遇攻擊）— 從加密端與解密端同時做較小的暴力搜尋並在中間比對;2DES 因此無效,有效強度僅約單一 DES（約 2^57）,故業界直接採 3DES。
Frequency Analysis（頻率分析）— 利用語言字母出現頻率（如英文 E 最常見）破解替換式密碼;現代密碼以統計隨機化反制。
Implementation / Fault Injection（實作攻擊/故障注入）— 攻擊實作弱點（程式錯誤、過時韌體）,或以高電壓、極端溫度等外部故障使密碼裝置失效。
ICS/SCADA（工業控制系統）— 控制實體流程，重可用與安全。
OT（營運技術）— Operational Technology，與 IT 對應。
Air Gap（實體隔離）— 網路完全隔絕（Stuxnet 證明非萬靈）。
06 — 易混淆釐清

暴力 vs 字典 vs 生日：暴力＝試所有可能；字典＝試常見值；生日＝攻擊雜湊碰撞（利用機率，所需嘗試遠少於暴力）。

旁通道打『實作』非『數學』：演算法理論安全，但實作洩漏時間/功耗仍可被破。對策是常數時間實作。

IT 的 CIA vs OT 的優先序：IT 常以機密性優先；OT/ICS 以可用性與人身安全(safety)優先——停機或誤動作可能危及生命。

07 — 條文與標準

NIST SP 800-82（ICS 安全）、IEC 62443（工控資安）、NIST SP 800-57（金鑰長度建議以抵抗暴力）。後量子密碼（PQC）因應量子電腦對 RSA/ECC 的威脅，NIST 2024 起發布標準（如 ML-KEM）。

08 — 真實事件：Stuxnet 2010
🔎 跨越 air gap 的工控攻擊

Stuxnet（2010）是針對伊朗核設施ICS/SCADA的精密惡意程式：透過 USB 跨越實體隔離（air gap），利用多個零時差漏洞，悄悄竄改離心機 PLC 的轉速使其損壞、同時對監控畫面回報『正常』。它證明了air gap 不是萬靈丹，且 OT 攻擊可造成實體破壞。

教訓：OT/ICS 安全攸關人身與實體安全；需縱深防禦（分段、USB 管控、韌體完整性、行為監控），不能只靠隔離。

📌 延伸：為什麼 IoT 這麼難防

IoT 裝置常預設弱密碼、無法更新、長期在線，成為殭屍網路溫床。Mirai（2016）就是掃描預設密碼的 IoT（攝影機/路由器）組成大型 DDoS（呼應 D4）。防禦要從採購要求（可更新/不可預設密碼）、網路分段、最小對外著手——把 IoT 當『不可信端點』隔離。

09 — CISSP 考點提醒

選最佳答案：『量測加密時間/功耗破金鑰』→ 旁通道；『攻擊雜湊碰撞』→ 生日攻擊；『ICS 最該優先保護』→ 可用性/安全(safety)；『跨 air gap』→ Stuxnet/USB。

常見陷阱：air gap 非絕對安全；OT 與 IT 的 CIA 優先序不同；旁通道打實作不打數學。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：密碼分析含暴力/字典/生日/旁通道/明文攻擊；旁通道打實作；IoT/ICS 重可用與 safety、需分段與完整性；Stuxnet 證明 air gap 非萬靈。

Q1 . 量測功耗/時間推測金鑰是什麼攻擊？
→ 旁通道攻擊（side-channel），打實作非數學。

Q2 . ICS/OT 系統的安全優先序與一般 IT 有何不同？
→ OT 以可用性與人身安全(safety)優先；IT 常以機密性優先。

Q3 . Stuxnet 證明了什麼防禦迷思？
→ air gap（實體隔離）並非絕對安全（可經 USB 跨越）。

🎯 實戰演練 EXTRA
QUESTION 1

To strengthen a legacy system, an architect proposes encrypting data twice with DES using two different 56-bit keys (2DES), claiming an effective strength of 112 bits. Why does the security reviewer reject this claim?

（為強化舊系統,架構師提議用兩把不同的 56-bit 金鑰對資料做兩次 DES 加密（2DES）,宣稱有效強度可達 112-bit。安全審查者為何否決此宣稱?）

A. A birthday attack can find hash collisions in 2DES 生日攻擊可找出 2DES 的雜湊碰撞
B. A meet-in-the-middle attack reduces the effective strength to roughly single DES 中間相遇攻擊使有效強度僅約等於單一 DES
C. Frequency analysis can defeat any substitution-based cipher 頻率分析可破解任何替換式密碼
D. A chosen-ciphertext attack can recover both keys directly 選擇密文攻擊可直接還原兩把金鑰
✅ 解答與解析

正解：B。Meet-in-the-middle 是典型的已知明文攻擊:攻擊者從明文端『加密』、密文端『解密』同時進行兩個較小的暴力搜尋,在中間比對結果——把 2DES 的有效強度打回約 2^57,幾乎等於單一 DES。這正是業界跳過 2DES、直接採用 3DES 的原因。記憶:兩次加密≠強度相加,中間會『相遇』。

為何其他選項錯：
  • A：生日攻擊針對『雜湊碰撞』機率,與區塊加密的金鑰強度無關。
  • C：頻率分析針對簡單替換式密碼,對 DES 這類現代區塊加密無效。
  • D：選擇密文是一種攻擊『模型』（可取得解密結果）,並不能直接還原 2DES 金鑰;2DES 的致命傷是 meet-in-the-middle。
QUESTION 2

A penetration tester finds a web service that encrypts any attacker-supplied input with the server's secret key and returns the resulting ciphertext. The tester submits carefully crafted inputs and studies the outputs to deduce information about the key. Which attack model does this BEST describe?

（滲透測試員發現某網路服務會用伺服器的密鑰加密攻擊者提交的任意輸入並回傳密文。測試員提交精心設計的輸入並分析輸出,以推導金鑰資訊。這最符合哪種攻擊模型?）

A. Ciphertext-only attack 唯密文攻擊
B. Known-plaintext attack 已知明文攻擊
C. Chosen-ciphertext attack 選擇密文攻擊
D. Chosen-plaintext attack 選擇明文攻擊
✅ 解答與解析

正解：D。攻擊者能『自選任意明文』並取得對應密文（一個加密 oracle）＝選擇明文攻擊（CPA）;強度排序:唯密文 < 已知明文 < 選擇明文 < 選擇密文。differential cryptanalysis 即屬 CPA。判斷關鍵:誰『選』了輸入?選的是明文還是密文?

為何其他選項錯：
  • A：唯密文攻擊手上只有密文,無法自選輸入。
  • B：已知明文是『被動取得』明密文對;本題攻擊者可主動選擇明文,更強。
  • C：選擇密文要能提交『密文』並取得『解密結果』;此服務只提供加密方向。
QUESTION 3

A critical vulnerability is announced for the PLCs controlling a manufacturing line. The vendor has released a patch, but the operations team warns that unplanned downtime could create safety hazards and halt production. What is the BEST course of action?

（控制產線的 PLC 被公告存在重大漏洞,廠商已釋出修補程式,但營運團隊警告非計畫性停機可能造成安全危害並中斷生產。最佳做法是什麼?）

A. Apply the patch to all PLCs immediately during production 立即在生產中對所有 PLC 套用修補
B. Rely on the existing air gap and skip the patch 依賴既有 air gap 而不修補
C. Replace the PLCs with general-purpose IT servers 以一般用途 IT 伺服器汰換 PLC
D. Apply compensating controls (segmentation, monitoring) now, and deploy the vendor-approved patch in a planned maintenance window 先實施補償性控制（分段/監控）,再於排定維護窗口套用廠商核可的修補
✅ 解答與解析

正解：D。OT/ICS 以可用性與人身安全（safety）優先:先以網路分段、強化監控等補償性控制降低暴露,再於排定的維護窗口套用『廠商核可』的修補（ICS 只用 vendor-approved patch）——同時管理資安風險與營運/安全風險,是 think-like-a-manager 的答案。

為何其他選項錯：
  • A：生產中貿然修補可能導致停機與安全事故,違反 OT 的優先序。
  • B：Stuxnet 已證明 air gap 非絕對安全,對重大漏洞不作為等於接受過高風險。
  • C：PLC 是特殊用途的工控設備,一般 IT 伺服器無法取代,且引入更大的變更風險。
📑 官方考綱對照重點（Official Objectives · §3.7 密碼攻擊 / §3.5 IoT/ICS）
▍密碼分析攻擊

Brute force（彩虹表/字典/混合）、Known-plaintext（Meet-in-the-middle→2DES 弱）、Chosen ciphertext（差分分析）、Side-channel/Timing（功耗/電磁/時間，TEMPEST 防護）、MITM、Pass-the-Hash（NTLM）、Kerberos Golden/Silver Ticket。

▍ICS 與 IoT

ICS＝OT，含 SCADA／PLC／DCS；防護靠網路分段、僅用廠商核可修補。IoT：獨立隔離網段（three dumb routers）、關閉遠端管理。

對應官方 §3.7 密碼攻擊、§3.5.5 ICS、§3.5.8 IoT。

§3.7.1 Brute force
• Brute force: an attack that attempts every possible valid combination for a key or password
• they involve using massive amounts of processing power to methodically guess the key used to secure cryptographic communications
• Rainbow Tables: large, precomputed databases that store chains of plaintext passwords and their corresponding cryptographic hash values, allowing an attacker to quickly look up a hash and retrieve the original plaintext password without having to compute hashes in real-time
• Dictionary Attack: a password-cracking method in which the attacker systematically tries every word in a dictionary or word list
• Hybrid Attack: is a password-cracking method that combines elements of both dictionary and brute-force attacks; better success rate compared to a pure dictionary attack, and faster than a full brute-force method
• Brute force countermeasures:
• strong password policies
• password hashing and salting
• key stretching
§3.7.10 Man-in-the-middle (MITM)
• Man-in-the-middle (MITM) (AKA on-path, or Adversary-in-the-middle(AitM)): in this attack a malicious individual sits between two communicating parties and intercepts all communications (including the setup of the cryptographic session)
• attacker responds to the originator's initialization requests and sets up a secure session with the originator
• attacker then establishes a second secure session with the intended recipient using a different key and posing as the originator
• attacker can then "sit in the middle" of the communication and read all traffic as it passes between the two parties
• Countermeasures: use a combination of strong protocols, vigilant practices, and robust authentication such as strong encryption
§3.7.11 Pass the hash
• Pass the hash (PtH): a technique where an attacker captures a password hash (as opposed to the password characters) and then simply passes it through for authentication and potentially lateral access to other networked systems
• the threat actor doesn’t need to decrypt the hash to obtain a plain text password
• PtH attacks exploit the authentication protocol, as the password's hash remains static for every session until the password is rotated
• attackers commonly obtain hashes by scraping a system's active memory and other techniques
• PtH attacks typically exploit NTLM vulns, but attackers also use similar attacks against other protocols, including Kerberos
• Countermeasures: defense-in-depth strategy that focuses on preventing initial access, securing credentials, and restricting lateral movement
§3.5.5 Industrial Control Systems (ICS)
• Industrial control systems (ICS): a form of computer-management device that controls industrial processes and machines, also known as operational technology (OT); there are several forms of ICS including distributed control systems (DCS), programmable logic controllers (PLC), and supervisory control and data acquisition (SCADA)
• recognize that DCS, PLC, and SCADA are types of ICS; and know about how to secure ICS
• Supervisory control and data acquisition (SCADA): systems used to control physical devices like those in an electrical power plant or factory; SCADA systems are well suited for distributed environments, such as those spanning continents
• some SCADA systems still rely on legacy or proprietary communications, putting them at risk, especially as attackers gain knowledge of such systems and their vulnerabilities
• ICS risk mitigations:
• network segmentation
• robust change management: document changes, and configurations; only apply vendor-approved patches
• physical security: controls should include physical access restrictions to components
• encryption: use secure communication between components
• system hardening/logging: disable unused services/ports, log and monitor activities
• DCS (Distributed Control System): a control system usually found within a single factory or plant; uses a network of controllers managed by a central supervisory control level to control local processes
• PLC (Programmable Logic Controller): a rugged, industrial computer used to automate specific electromechanical processes
§3.5.8 Internet of Things (IoT)
• Internet of things (IoT): a class of smart devices that are internet-connected in order to provide automation, remote control, or AI processing to appliances or devices
• An IoT device is almost always separate/distinct hardware used on its own or in conjunction with an existing system
• IoT security concerns often relate to access and encryption
• IoT is often not designed with security as a core concept, resulting in security breaches; once an attacker has remote access to the device they may be able to pivot
• Securing IoT:
• deploy a distinct network for IoT equipment, kept separate and isolated (known as three dumb routers)
• keep systems patched
• limit physical and logical access
• monitor activity
• implement firewalls and filtering
• never assume IoT defaults are good enough, evaluate settings and config options, and make changes to optimize security while supporting business functions
• disable remote management and enable secure communication only (such as over HTTPS)
• review IoT vendor to understand their history with reported vulnerabilities, response time to vulnerabilities and their overall approach to security
• not all IoT devices are suitable for enterprise networks
