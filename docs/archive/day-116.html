考點：SDLC 與安全活動對應

Exam: SDLC & Security Activities
01 — 考前重點濃縮

D8 後期。本封考SDLC 各階段對應的安全活動(最常考)。把『哪個活動在哪個階段』背熟。

需求→安全/隱私需求;設計→威脅建模;開發→安全編碼+SAST;測試→DAST/滲透;部署/維運→組態/監控/修補。
越早處理漏洞越便宜(shift left)。
威脅建模屬設計階段。
02 — 階段對應速記
階段安全活動
需求安全/隱私需求、濫用案例
設計威脅建模(STRIDE)、架構審查
開發安全編碼、SAST、同儕審查
測試DAST、滲透、誤用案例
部署/維運安全組態、監控、修補
03 — 關鍵名詞（中英）
Threat Modeling（威脅建模）— 設計階段找威脅。
Shift Left（安全左移）— 越早處理越便宜。
Secure Coding（安全編碼）— 開發階段。
Abuse Case（濫用案例）— 需求/設計期的攻擊者視角。
SAST/DAST（靜/動態測試）— 開發/測試階段。
04 — 常見陷阱

威脅建模在設計階段(不是測試)。

越早抓漏洞越便宜;別把安全留到上線前。

SAST(開發/讀碼) vs DAST(測試/打站)。

05 — 🎯 實戰考題
QUESTION 1

At which SDLC phase is threat modeling MOST effectively performed?

（威脅建模在 SDLC 哪個階段執行最有效?）

A. Requirements 需求
B. Design 設計
C. Testing 測試
D. Deployment 部署
✅ 解答與解析

正解：B。威脅建模在設計階段最有效——此時系統架構成形、變更成本最低,可系統化找出威脅並調整設計。

為何其他選項錯：
  • A：需求期定安全需求,但威脅建模需有設計可分析。
  • C：測試期才做威脅建模已太晚(設計已定)。
  • D：部署期更晚,修改成本最高。
QUESTION 2

When is it MOST cost-effective to address a security flaw in the SDLC?

（在 SDLC 中,何時處理安全缺陷最具成本效益?）

A. As early as possible (requirements/design) 越早越好(需求/設計)
B. During testing 測試期
C. After deployment 上線後
D. Only when exploited 被利用時才處理
✅ 解答與解析

正解：A。越早(需求/設計)處理成本越低(shift left);上線後修復成本可能高出數十倍。

為何其他選項錯：
  • B：測試期已比設計期貴。
  • C：上線後修復最貴(可能召回/重構)。
  • D：被利用才處理已造成損害。
QUESTION 3

Static Application Security Testing (SAST) is typically integrated at which phase?

（靜態應用安全測試(SAST)通常整合在哪個階段?）

A. Requirements 需求
B. Development (as code is written) 開發(撰寫程式碼時)
C. Deployment 部署
D. Decommissioning 除役
✅ 解答與解析

正解：B。SAST 分析原始碼,整合在開發階段(撰寫/提交程式碼時),能即早抓到根因(shift left)。

為何其他選項錯：
  • A：需求期尚無程式碼。
  • C：部署期 SAST 已太晚。
  • D：除役與 SAST 無關。
06 — (ISC)² 選最佳答案思路

威脅建模=設計;SAST=開發;DAST/滲透=測試;監控/修補=維運。

越早處理漏洞越便宜(shift left);看到『最具成本效益的時機』選最早階段。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：SDLC 各階段對應安全活動:設計→威脅建模、開發→SAST/安全編碼、測試→DAST;越早越便宜。

Q1 . 威脅建模最有效的階段？
→ 設計階段。

Q2 . 處理安全缺陷最具成本效益的時機？
→ 越早越好(需求/設計,shift left)。

Q3 . SAST 整合在哪個階段？
→ 開發階段(撰寫程式碼時)。
