DevSecOps 與 CI/CD 安全

DevSecOps & CI/CD Security
01 — DevSecOps 與 CI/CD 安全

DevOps 講求快速、自動化交付；DevSecOps 把安全織進這條自動化管線——『安全即程式碼、人人有責』。你熟悉 GitHub Actions，今天看 CI/CD 管線本身的安全風險與防護。

🗺️ 今天重點：DevOps→DevSecOps → CI/CD 管線風險 → 管線安全控制 → 環境隔離。
02 — CI/CD 管線的安全風險
風險說明
管線投毒竄改 build 腳本/相依，植入惡意碼到產物
祕密外洩API key/憑證硬編在 repo 或 log
相依污染引入惡意/有漏洞的第三方套件
過度權限CI runner/token 權限過大，被濫用
03 — 管線安全控制
管線中嵌入 SAST/SCA/secret scanning，未過不得部署
最小權限 token、短期憑證、祕密用 vault 管理（不入 repo）
簽章與來源可驗（artifact signing、SLSA、SBOM）
環境隔離：開發/測試/正式分離，正式環境變更受控
04 — 關鍵名詞速查
DevSecOps（開發安全維運一體）— 把安全自動化織入 CI/CD。
CI/CD（持續整合/持續交付）— 自動化建置、測試、部署管線。
Pipeline Poisoning（管線投毒）— 竄改建置流程植入惡意碼。
Secret Management（祕密管理）— 以 vault 管理憑證，不硬編入碼。
SBOM（軟體物料清單）— 列出所有相依元件，利於漏洞追蹤。
Environment Segregation（環境隔離）— 開發/測試/正式環境分離。
05 — 易混淆釐清

DevOps vs DevSecOps：DevOps 求快與自動化；DevSecOps 在其中內建安全自動化（安全是管線的一部分，不是事後關卡）。

祕密管理：把金鑰寫進程式或 CI 變數明文是常見錯誤；應用祕密庫＋短期憑證並輪替。

06 — 條文與標準

NIST SSDF（SP 800-218）、SLSA（供應鏈完整性等級）、OWASP CI/CD Top 10。SBOM 受 美國 EO 14028 推動，用於軟體供應鏈透明度。

07 — 真實事件：管線投毒 / 供應鏈
🔎 攻擊建置流程，一次毒害所有下游

攻擊者鎖定 CI/CD 與軟體供應鏈：竄改建置流程或熱門套件，讓正常簽署的更新夾帶後門，一次散布到大量下游（SolarWinds 2020 即為建置流程被植入的經典案例）。因為產物『看似官方』，極難察覺。

教訓：保護建置環境＝保護所有下游；需 artifact 簽章、來源可驗（SLSA）、最小權限、SBOM 追蹤相依。

08 — CISSP 考點提醒

選最佳答案：『把安全融入自動化交付』→ DevSecOps；『防 CI 祕密外洩』→ 祕密庫＋最小權限 token；『軟體供應鏈透明』→ SBOM。

常見陷阱：安全不是 CI/CD 最後一道手動關卡，而是自動化內建；憑證勿硬編入 repo。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：DevSecOps 把安全自動化織入 CI/CD；管線風險含投毒/祕密外洩/相依污染；對策是掃描關卡、祕密庫、簽章與環境隔離。

Q1 . CI 中防止憑證外洩的做法？
→ 用祕密庫管理＋短期最小權限 token，不硬編入 repo。

Q2 . 用來追蹤軟體所有相依元件的清單？
→ SBOM（軟體物料清單）。

Q3 . 為何攻擊者鎖定建置流程？
→ 一次污染可散布到所有下游，且產物看似官方、難察覺。

📑 官方考綱對照重點（Official Objectives · §8.2.6 CI/CD / §8.1.1 DevSecOps）
▍CI/CD

CI 持續整合：程式一寫好即併入共享 repo、觸發單元/整合測試、及早抓錯。CD：Continuous Delivery 持續交付＝隨時可部署但部署需人工核可；Continuous Deployment 持續部署＝過測即自動上線無需人工。

▍DevSecOps

把安全左移 shift-left整進 CI/CD pipeline；用威脅建模、最小權限、零信任保護常依賴第三方工具的 pipeline。

對應官方 §8.2.6 CI/CD、§8.1.1 DevSecOps。

§8.2.6 Continuous Integration and Continuous Delivery (CI/CD)
• Continuous Integration and Continuous Delivery: workflow automation processes and tools that help reduce, if not eliminate, the need for manual communication and coordination between the steps of a software development process
• Continuous integration (CI): all new code is integrated into the rest of the system as soon as the developer writes it, merging it into a shared repo
• this merge triggers a batch of unit tests
• if it merges without error, it's subjected to integration tests
• CI improves software development efficiency by identifying errors early and often
• CI also allows the practice of continuous delivery (CD)
• Continuous Delivery (CD): incrementally building a software product that can be released at any time; because all processes and tests are automated, code can be released to production daily or more often
• Note that continuous delivery and continuous deployment are both automation-heavy DevOps practices that build on CI; the key difference is in the final step of the release process: human intervention
• Continuous Delivery: code is always in a deployable state, but deployment to production requires a manual approval step
• Continuous Deployment: every change that passes automated testing is automatically deployed to production with no manual intervention
• CI/CD relies on automation and often third-party tools which can have vulnerabilities or be compromised
• Secure practices such as threat modeling, least privilege, defense in depth, and zero trust can help reduce possible threats to these tools and systems
