考點：OWASP 漏洞辨識

Exam: OWASP Vuln Identification
01 — 考前重點濃縮

本封考OWASP 常見漏洞辨識與防禦:注入/XSS/CSRF。從情境判斷是哪種漏洞、用什麼防。

注入→參數化查詢;XSS→輸出編碼/CSP;CSRF→token/SameSite。
XSS=在受害者瀏覽器執行腳本;CSRF=冒用身分送請求。
輸入用白名單驗證;前端限制不是安全控制。
02 — 漏洞速記
漏洞防禦
SQL 注入參數化查詢、最小權限
XSS輸出編碼、CSP、HttpOnly
CSRFCSRF token、SameSite cookie
越權(BOLA)伺服器端每物件授權
03 — 關鍵名詞（中英）
Injection（注入）— 輸入被當指令執行。
XSS（跨站腳本）— 在他人瀏覽器執行腳本。
CSRF（跨站請求偽造）— 冒用已登入身分送請求。
Output Encoding（輸出編碼）— 防 XSS 主力。
Parameterized Query（參數化查詢）— 防注入根本解。
04 — 常見陷阱

XSS(執行腳本) vs CSRF(冒用請求)防法不同。

防 XSS 主力是輸出編碼(非僅輸入驗證)。

防注入根本是參數化(非黑名單過濾)。

05 — 🎯 實戰考題
QUESTION 1

An attacker enters ' OR '1'='1 into a login field and bypasses authentication. Which vulnerability is this, and what is the BEST fix?

（攻擊者在登入欄輸入 ' OR '1'='1 而繞過驗證。這是哪種漏洞,最佳修法為何?）

A. XSS; output encoding XSS;輸出編碼
B. SQL injection; parameterized queries SQL 注入;參數化查詢
C. CSRF; anti-CSRF tokens CSRF;CSRF token
D. Buffer overflow; ASLR 緩衝區溢位;ASLR
✅ 解答與解析

正解：B。輸入被當 SQL 執行而繞過登入＝SQL 注入;最佳根本修法是參數化查詢(預備語句)讓資料不被當指令。

為何其他選項錯：
  • A：XSS 是注入腳本到頁面,非此 SQL 情境。
  • C：CSRF 是冒用請求,非注入。
  • D：緩衝區溢位是記憶體問題,非 SQL 注入。
QUESTION 2

A forum stores a malicious script in a comment; when other users view it, the script runs in their browsers and steals session cookies. This is:

（某論壇將惡意腳本存入留言;其他使用者瀏覽時腳本在其瀏覽器執行並竊取 session cookie。這是:）

A. CSRF
B. Stored XSS 儲存型跨站腳本
C. SQL injection
D. Clickjacking
✅ 解答與解析

正解：B。腳本被存入並在他人瀏覽時於其瀏覽器執行＝Stored XSS;防禦:輸出編碼+CSP+HttpOnly cookie。

為何其他選項錯：
  • A：CSRF 是冒用身分送請求,不執行腳本。
  • C：SQL 注入針對資料庫,非瀏覽器腳本。
  • D：點擊劫持是誘導點擊隱藏元素,非腳本注入。
QUESTION 3

Which defense BEST mitigates Cross-Site Request Forgery (CSRF)?

（哪個防禦最能緩解跨站請求偽造(CSRF)?）

A. Parameterized queries 參數化查詢
B. Anti-CSRF tokens and SameSite cookies CSRF token 與 SameSite cookie
C. Disk encryption 磁碟加密
D. Rate limiting 速率限制
✅ 解答與解析

正解：B。CSRF 緩解靠 CSRF token(請求需帶不可預測的 token)＋SameSite cookie(限制跨站送 cookie)。

為何其他選項錯：
  • A：參數化查詢防注入,非 CSRF。
  • C：磁碟加密與 CSRF 無關。
  • D：速率限制防濫用/暴力,非 CSRF。
06 — (ISC)² 選最佳答案思路

注入→參數化;XSS→輸出編碼/CSP;CSRF→token/SameSite。先辨識漏洞型態再配對防禦。

XSS(執行腳本) vs CSRF(冒用請求)別混;前端限制非安全控制。

✏️ 一句話總結 + 自我檢測（點開看解答）

一句話：注入(參數化)、XSS(輸出編碼/CSP/HttpOnly)、CSRF(token/SameSite);辨識型態配對防禦。

Q1 . ' OR '1'='1 繞過登入是什麼漏洞、怎麼根治？
→ SQL 注入;參數化查詢。

Q2 . 留言中腳本在他人瀏覽器執行竊 cookie 是？
→ Stored XSS。

Q3 . 防 CSRF 的主要手段？
→ CSRF token 與 SameSite cookie。

🎯 實戰演練 EXTRA
QUESTION 4

A tester logged in as a regular customer changes the URL from /orders?id=1001 to /orders?id=1002 and views another customer's order. Which vulnerability is this, and what is the BEST remediation?

（測試者以一般客戶身分登入,把網址 /orders?id=1001 改成 /orders?id=1002 就看到他人訂單。這是哪種漏洞,最佳修法為何?）

A. CSRF; anti-CSRF tokens CSRF;CSRF token
B. SQL injection; parameterized queries SQL 注入;參數化查詢
C. Session hijacking; enforce TLS 工作階段劫持;強制 TLS
D. Broken access control (IDOR); server-side per-object authorization checks 存取控制失效(IDOR);伺服器端逐物件授權檢查
✅ 解答與解析

正解：D。改個參數就能讀他人資料＝Broken Access Control(IDOR/BOLA):應用只驗證『已登入』卻沒驗證『此使用者有權存取此物件』。修法:每次請求都在伺服器端做逐物件授權檢查——前端隱藏或猜不到的 ID 都不是授權控制。

為何其他選項錯：
  • A：攻擊者用的是自己的合法 session 直接存取,並非冒用他人身分送請求。
  • B：輸入未被當 SQL 執行,只是改了資源識別碼。
  • C：沒有竊取他人 session;TLS 保護傳輸,無法補授權缺失。
QUESTION 5

A newly deployed web server still uses the vendor's default administrator password, has directory listing enabled, and returns detailed stack traces to users. Which OWASP Top 10 category BEST describes this condition?

（新部署的網頁伺服器仍使用原廠預設管理者密碼,開啟目錄列示,並向使用者回傳詳細堆疊追蹤。哪個 OWASP Top 10 類別最能描述此狀況?）

A. Security misconfiguration 安全組態錯誤
B. Insecure design 不安全設計
C. Authentication failures 鑑別失效
D. Cryptographic failures 密碼學失效
✅ 解答與解析

正解：A。預設帳密,目錄列示,詳細錯誤訊息——共同根因是部署設定未強化＝Security Misconfiguration。修法:強化基準(hardening baseline),變更所有預設值,關閉不必要功能。判斷邏輯:多個症狀指向同一根因時,選涵蓋根因的類別。

為何其他選項錯：
  • B：設計本身沒有缺陷,是部署組態問題(design flaw＝就算都做對仍有漏洞,此處改設定即可修)。
  • C：預設密碼是誘餌——根因是『沒改預設組態』,非鑑別機制本身失效。
  • D：與加密保護無關。
QUESTION 6

A web application was breached three months ago, but the intrusion was only discovered when a customer complained. Investigation shows the application recorded no authentication failures or access anomalies at all. Which OWASP Top 10 category BEST describes the root weakness?

（某網頁應用三個月前遭入侵,直到客戶投訴才被發現;調查顯示應用完全沒有記錄任何鑑別失敗或存取異常。根本弱點屬於哪個 OWASP 類別?）

A. Insecure design 不安全設計
B. Security logging and alerting failures 安全記錄與警示失效
C. Authentication failures 鑑別失效
D. Security misconfiguration 安全組態錯誤
✅ 解答與解析

正解：B。入侵三個月無人知,因為應用完全沒有記錄與警示＝OWASP Security logging and alerting failures:沒有 log 就沒有偵測,回應與鑑識能力。判斷邏輯:題幹問的不是『怎麼被入侵』,而是『為何這麼久才發現』——弱點在可觀測性。

為何其他選項錯：
  • A：問題不在功能設計,而在偵測能力缺失。
  • C：攻擊者或許利用了鑑別弱點,但題幹指的根本弱點是『看不見』。
  • D：記錄缺失有專屬類別,比籠統的組態錯誤更精確(選最佳答案)。

▍情境辨識

使用者輸入直接拼進 SQL→Injection/SQLi；回顯未編碼的使用者輸入→XSS；冒用已登入者送請求→CSRF；越權存取他人資料→Broken Access Control；預設帳密/開放設定→Security Misconfiguration。

先辨識漏洞類型，再選對應防禦（參數化/編碼/token/授權檢查）。

§8.5.1 Security weaknesses and vulnerabilities at the source-code level
• A source code vulnerability is a code defect providing a threat actor with an opportunity to compromise the security of a software system
• source code vulns are caused by design or implementation flaws
• design flaw: if dev did everything correctly, there would still be a vulnerability
• implementation flaw: dev incorrectly implemented part of a good design
• the OWASP top 10 vulnerabilities (2025 edition):
• Broken access control
• Security misconfiguration
• Software supply chain failures
• Cryptographic failures
• Injection
• Insecure design
• Authentication failures
• Software or data integrity failures
• Security logging and alerting failures
• Mishandling of exceptional conditions
